# Настройка интеграций: инструкция для разработчика

Эта инструкция — для того, **кто технически подключает приём заявок**: верстает форму на сайте,
настраивает плагин WordPress или пишет код, отправляющий данные по API. Администратор компании в
Лид-Канале выдал вам одно из двух:

- **ссылку (URL)** — для Tilda или WordPress;
- **API-ключ** — для любого другого сайта или сервиса (универсальный webhook).

Ниже — что именно делать с тем, что вам передали, по каждому из трёх вариантов, плюс общие правила
формата данных и разбор ошибок. Раздел большой — пользуйтесь оглавлением справа, чтобы перейти к
нужному источнику.

> 💡 Если ваш сайт не Tilda и не WordPress, но умеет сам отправлять POST-запрос — это раздел
> «Универсальный webhook» ниже, а не Tilda/WordPress.

---

## Как это устроено — коротко

Приём заявок — это всегда **один и тот же принцип**, независимо от источника:

```
Ваш сайт/сервис  →  POST-запрос с данными формы  →  Лид-Канал  →  заявка появилась в CRM
```

Разница между тремя вариантами — только в том, **как система узнаёт, какой компании принадлежит
заявка**:

| Источник | Как определяется компания | Что вам передали |
| --- | --- | --- |
| **Tilda** | зашита прямо в URL (`.../tilda/<id-компании>`) | готовая ссылка, просто вставить |
| **WordPress** | зашита прямо в URL (`.../wordpress/<id-компании>`) | готовая ссылка, просто вставить |
| **Универсальный webhook** | по API-ключу в запросе | ключ, который вы сами кладёте в запрос |

Во всех трёх случаях данные распознаются одним и тем же кодом: система сама раскладывает
известные поля (имя, телефон, email, комментарий, UTM-метки) по нужным местам, а всё незнакомое —
сохраняет как есть, ничего не теряя. Подробности — в разделе [«Какие поля распознаются»](#какие-поля-формы-распознаются-автоматически).

> ⚠️ **Заявка не теряется почти никогда.** Даже если поля формы называются нестандартно, заявка
> всё равно попадёт в систему — просто нераспознанные поля лягут в блок «Дополнительные поля»
> карточки лида, а не в имя/телефон/email. Хуже, чем «заявка сохранилась не в тех полях», быть не
> должно — «заявка не сохранилась вообще» в норме не бывает (единственное исключение — выключенный
> тумблером источник, см. [коды ошибок](#коды-ответа-и-ошибки)).

## Tilda

Вам передали ссылку вида:

```
https://<домен-лид-канала>/api/webhooks/tilda/<id-компании>
```

### Что с ней сделать

1. Откройте нужную форму в редакторе Tilda.
2. В настройках формы найдите раздел приёма данных на другие системы — в интерфейсе Tilda это
   обычно называется **«CRM и системы аналитики»** → **Webhook** (точное название пункта может
   отличаться в зависимости от версии редактора Tilda — ищите слово «Webhook»).
3. Вставьте туда ссылку **целиком, без изменений**. Метод — `POST` (обычно выставлен по умолчанию,
   менять не нужно).
4. Сохраните форму и опубликуйте страницу.

### Как Tilda присылает тестовый запрос

У Tilda есть встроенная кнопка «Отправить тестовые данные» (или аналогичная) в настройках вебхука.
Она шлёт запрос с полем `test=test` — Лид-Канал отвечает `200 OK`, но **заявку по этому запросу не
создаёт**. Это ожидаемое поведение, не ошибка: тестовый пинг Tilda предназначен только для проверки
доступности адреса, а не для проверки реального приёма.

Чтобы проверить приём по-настоящему — заполните и отправьте саму форму на опубликованной странице
(см. [«Как проверить, что всё заработало»](#как-проверить-что-всё-заработало)).

### Имена полей в Tilda

Поля, которые Tilda присылает по умолчанию (`name`, `phone`, `email` или их русские подписи «Имя»,
«Телефон», «Email»), распознаются автоматически — ничего дополнительно настраивать не нужно.
Если в форме есть свои дополнительные поля — они сохранятся в заявке под своими именами
(раздел «Дополнительные поля» карточки лида).

## WordPress

Вам передали ссылку вида:

```
https://<домен-лид-канала>/api/webhooks/wordpress/<id-компании>
```

### Важно понимать сразу

WordPress **сам по себе** не умеет отправлять webhook — это делает плагин формы или отдельный плагин
для вебхуков. Какой именно плагин используется на сайте, решаете вы; вот три рабочих варианта:

| Плагин форм | Как отправить вебхук |
| --- | --- |
| **Gravity Forms** | встроенный официальный **Webhooks Add-On** — добавляете «Feed» с URL, метод `POST` |
| **Fluent Forms** | встроенная функция **Webhook** в настройках формы (не нужен отдельный плагин) |
| **Contact Form 7** | сам по себе вебхук не умеет — поставьте отдельный плагин, например **WP Webhooks** (универсальный, работает поверх CF7/Elementor Forms и др.) |
| **WPForms** | вебхук есть в платных тарифах (**Webhooks addon**); на бесплатном — тот же путь через **WP Webhooks** |

Какой бы плагин вы ни выбрали — везде нужно одно и то же:

1. Вставить ссылку выше в поле URL вебхука.
2. Метод запроса — `POST`.
3. Формат — на ваш выбор, JSON или обычная форма (`x-www-form-urlencoded`): Лид-Канал распознаёт оба
   автоматически по заголовку `Content-Type`, отдельно настраивать не нужно.
4. Никаких дополнительных заголовков (авторизация, токены) не требуется — компания уже определена
   URL-ом.

### Имена полей в WordPress

Если можете задать имена полей формы сами — используйте `name`, `phone`/`tel`, `email`, `comment`
(или их русские варианты «имя», «телефон», «email», «комментарий») — тогда они лягут сразу в
стандартные поля заявки.

> 💡 **Contact Form 7 — приятный бонус.** Стандартные имена полей CF7 (`your-name`, `your-email`,
> `your-tel`, `your-message`) распознаются автоматически «из коробки» — если вы не переименовывали
> поля вручную, ничего дополнительно настраивать не нужно.

Если плагин генерирует имена полей сам и поменять их нельзя (частый случай у конструкторов вроде
Elementor Forms) — не страшно: такие поля попадут в «Дополнительные поля» карточки лида под своим
техническим именем, заявка всё равно сохранится целиком.

## Универсальный webhook (API-ключ)

Этот вариант — для **любого сайта или сервиса**, который не Tilda и не WordPress: конструктор,
самописный сайт, CRM-коннектор, Zapier/Make-сценарий и т.п.

Вам передали не ссылку, а **API-ключ** — строку из 64 символов (цифры и латинские буквы), например:

```
a1b2c3d4e5f6...(64 символа всего)
```

> ⚠️ Ключ показывается администратору в Лид-Канале **только один раз**, при создании. Если ключ
> потерян — попросите администратора выпустить новый (старый при этом можно отключить или удалить в
> «Интеграциях»), никто, включая нас, не может «посмотреть» старый ключ заново.

### Адрес (один и тот же для всех компаний)

В отличие от Tilda/WordPress, у универсального webhook **нет своей ссылки на компанию** — конечная
точка одна для всех, а компания определяется по самому ключу:

```
POST https://<домен-лид-канала>/api/webhooks/leads
```

Уточните у администратора, на каком домене развёрнут Лид-Канал — обычно это тот же адрес, на который
он заходит в систему.

### Как передать ключ

Два способа — выберите тот, который поддерживает ваш инструмент:

**1. Заголовок (рекомендуется):**

```
X-Api-Key: <ваш-ключ>
```

**2. Параметр в адресе (если инструмент не умеет добавлять заголовки):**

```
POST https://<домен-лид-канала>/api/webhooks/leads?key=<ваш-ключ>
```

> ⚠️ **Никогда не кладите ключ в код, который выполняется в браузере посетителя** (JS прямо на
> странице сайта). Любой человек сможет открыть исходный код страницы и увидеть ключ — а значит,
> сможет слать заявки от имени вашей компании. Отправляйте запрос **с сервера** (PHP-скрипт,
> Node.js-бэкенд, серверная функция вашей платформы, сценарий Zapier/Make и т.п.), а не напрямую из
> браузера.

### Формат тела запроса

JSON или `x-www-form-urlencoded` — оба поддерживаются, выбирайте что удобнее. Заголовок
`Content-Type` желательно указывать явно и в соответствии с телом запроса.

**Пример — curl (JSON):**

```bash
curl -X POST "https://<домен-лид-канала>/api/webhooks/leads" \
  -H "X-Api-Key: <ваш-ключ>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Иван Иванов",
    "phone": "+7 900 123-45-67",
    "email": "ivan@example.com",
    "comment": "Интересует расчёт стоимости",
    "utm_source": "yandex",
    "utm_campaign": "leto2026"
  }'
```

**Пример — JS (Node.js/серверный код, не браузер):**

```js
await fetch('https://<домен-лид-канала>/api/webhooks/leads', {
  method: 'POST',
  headers: {
    'X-Api-Key': '<ваш-ключ>',
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    name: 'Иван Иванов',
    phone: '+7 900 123-45-67',
    email: 'ivan@example.com',
    comment: 'Интересует расчёт стоимости',
  }),
});
```

**Пример — PHP:**

```php
$ch = curl_init('https://<домен-лид-канала>/api/webhooks/leads');
curl_setopt($ch, CURLOPT_POST, true);
curl_setopt($ch, CURLOPT_HTTPHEADER, [
    'X-Api-Key: <ваш-ключ>',
    'Content-Type: application/json',
]);
curl_setopt($ch, CURLOPT_POSTFIELDS, json_encode([
    'name' => 'Иван Иванов',
    'phone' => '+7 900 123-45-67',
    'email' => 'ivan@example.com',
]));
curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);
$response = curl_exec($ch);
```

Успешный ответ — `200 OK`, тело `{"ok": true}`. Полный разбор ошибок — ниже.

## Какие поля формы распознаются автоматически

Не важно, через какой из трёх источников пришли данные — распознавание работает одинаково.

- **Заголовки полей регистронезависимы** (`Phone`, `PHONE`, `phone` — одно и то же), можно на
  русском или английском.
- Любое **неизвестное поле не отбрасывается** — сохраняется в «Дополнительных полях» карточки лида
  под своим оригинальным именем.

| Стандартное поле лида | Какие имена полей распознаются |
| --- | --- |
| Имя | `name`, `имя`, `фио`, `название`, `full_name`, `fullname`, `contact_name`, `клиент`, `заявитель`, `first_name`, `firstname`, `fname` (+ отдельное поле фамилии: `last_name`, `lastname`, `фамилия`, `surname` — допишется через пробел) |
| Телефон | `phone`, `телефон`, `tel`, `тел`, `telephone`, `mobile`, `phone_number`, `phonenumber`, `мобильный` |
| Email | `email`, `e-mail`, `почта`, `mail`, `email_address`, `emailaddress` |
| Комментарий | `comment`, `комментарий`, `comments`, `message`, `сообщение`, `text`, `текст`, `question`, `вопрос`, `заявка`, `описание`, `description` |
| UTM-метки | любое поле, начинающееся с `utm_` (`utm_source`, `utm_campaign`, ...) — попадает в блок маркетинговых данных лида целиком |
| Рекламные идентификаторы | `gclid`, `yclid`, `fbclid`, `msclkid`, `roistat`, `_ga` и другие метки трекинга — тоже в маркетинговые данные, не в «Дополнительные поля» |

> 📌 Если у вас на сайте нужны специальные скрытые поля для рекламы Яндекса (`yclid`, `client_id`) —
> это отдельная инструкция: **[«Поля для рекламы на сайте»](05-ad-tracking-fields.md)**.

> 💡 **Если два похожих поля пришли в одной заявке** (например, и `name`, и `фио`) — используется
> первое распознанное значение, второе не перезаписывает его. На практике это редкость: обычно в
> форме только один вариант имени поля.

## Коды ответа и ошибки

Любой запрос возвращает JSON и один из кодов ниже.

| Код | Тело ответа | Что значит | Что делать |
| --- | --- | --- | --- |
| `200` | `{"ok": true}` | Заявка принята и создана | всё в порядке |
| `200` | `{"ok": true}` | Это был тестовый пинг Tilda (`test=test`) — заявка **не создана** | ожидаемо, не ошибка |
| `401` | `{"error": "UNAUTHORIZED"}` | Только у универсального webhook: ключ не передан или неверный | проверьте заголовок `X-Api-Key`/параметр `key`, сверьте ключ с админом |
| `403` | `{"error": "SOURCE_DISABLED"}` | Источник выключен тумблером на странице «Интеграции» в CRM | попросите администратора включить источник |
| `404` | `{"error": "Company not found"}` | Только у Tilda/WordPress: в ссылке неверный или устаревший id компании | скопируйте ссылку заново со страницы «Интеграции» |
| `429` | `{"error": "RATE_LIMIT"}` | Слишком много запросов подряд (лимит — 60 запросов в минуту с одного адреса/ключа) | сбавьте частоту отправки, повторите позже |
| `500` | `{"error": "INTERNAL_ERROR"}` | Внутренняя ошибка на стороне Лид-Канала — заявка не сохранена | повторите запрос позже; если повторяется — напишите администратору или в поддержку |

> ⚠️ **`403 SOURCE_DISABLED` — не баг.** Это значит, что администратор компании сознательно выключил
> этот источник тумблером на странице «Интеграции» («у меня сейчас нет такого канала заявок»).
> Заявка в этом случае действительно не сохраняется — единственный такой случай во всей системе.
> Если вы не в курсе, почему источник выключен — уточните у администратора компании, прежде чем
> считать это технической проблемой на вашей стороне.

## Как проверить, что всё заработало

1. Отправьте тестовую заявку — реальную (не встроенный тестовый пинг Tilda, он не создаёт лид).
2. Откройте в Лид-Канале раздел **Интеграции** (`/admin/integrations`) — под карточкой источника
   должен появиться индикатор 🟢 «Активен» вместо ⚪ «Не настроено».
3. Откройте **Лиды** — новая заявка должна быть видна в списке, со всеми полями, которые вы отправили.

> 🖼️ **Скриншот — `images/06-dev-lead-customfields.png`**
> Карточка тестового лида: видно, как нераспознанные поля из тестового запроса легли в блок
> «Дополнительные поля».

> 💡 Если вы отправляете несколько тестовых заявок подряд с одинаковым телефоном/email — это не
> ошибка. Система не блокирует и не объединяет похожие заявки, а просто помечает их как возможный
> дубль — каждая всё равно сохраняется отдельной карточкой.

## Частые ошибки

| Ошибка | Почему не работает |
| --- | --- |
| Ссылку Tilda/WordPress скопировали не полностью (обрезали id компании в конце) | Запрос уйдёт на несуществующую компанию → `404 Company not found`. Скопируйте ссылку заново со страницы «Интеграции». |
| API-ключ вставили в JS-код на странице сайта | Ключ виден всем в исходном коде — уязвимость. Отправляйте запрос только с сервера. |
| Заголовок `X-Api-Key` не дошёл (некоторые конструкторы форм не дают добавлять свои заголовки) | Используйте способ через `?key=` в адресе вместо заголовка. |
| Ждут, что тестовая кнопка Tilda создаст лид | Тестовый пинг Tilda (`test=test`) намеренно не создаёт заявку — это только проверка доступности адреса. Присылайте реальную заявку через саму форму. |
| Заявка ушла, но полей в карточке лида не видно там, где ожидали | Имя поля не входит в список распознаваемых (см. таблицу выше) — значение всё равно сохранено, просто в блоке «Дополнительные поля», а не в «Имя»/«Телефон». Не потеря данных, а просто другое место в карточке. |
| Заявки вообще перестали приходить, хотя раньше приходили | Проверьте тумблер источника на странице «Интеграции» — возможно, администратор его выключил (`403 SOURCE_DISABLED`). |
| Отправляете заявки очень часто (нагрузочный тест, скрипт в цикле) | Лимит — 60 запросов в минуту с одного IP/ключа, дальше — `429 RATE_LIMIT`. Это защита от случайной перегрузки, не баг. |
