# Инструкция: что сделать руками (Task 1, Яндекс Метрика)

Это не про код. Это про то, что можете сделать **только вы** — я не могу зарегистрировать приложение на Яндексе или получить доступ к счётчику Метрики за вас (см. `TASK.md`).

Важно: это **новое, отдельное** OAuth-приложение — не то же самое, что уже зарегистрировано для Яндекс Директа (Phase 22). У Метрики свой `client_id`/`client_secret`/scope, и счётчик Метрики нередко живёт под другим аккаунтом Яндекса, чем рекламный кабинет Директа — так и должно быть.

Итог этих шагов: у меня будут `client_id`/`client_secret` и данные тестового счётчика для research, а вы в конце примете решение **go/no-go** по фазе.

---

## Шаг 1. Зарегистрировать OAuth-приложение на Яндексе

1. Открыть https://oauth.yandex.ru/client/new
2. **Название приложения** — любое понятное, например `Лид-Канал — Яндекс Метрика`.
3. **Платформы** → выбрать «Веб-сервисы».
4. **Redirect URI** — добавить:
   ```
   http://localhost:3000/api/integrations/yandex/metrika/callback
   ```
   Продовый домен можно дописать туда же позже — Яндекс позволяет отредактировать список Redirect URI в настройках приложения в любой момент, без пересоздания приложения.

   ⚠️ Этого роута в коде ещё нет — появится в Таске 2. Сейчас просто регистрируем адрес.
5. **Доступы** → в поле поиска ввести «Метрика» → отметить право:
   ```
   metrika:offline_data — загрузка в счётчики офлайн-данных (данные из CRM, офлайн-конверсии, звонки)
   ```
   Это именно то право, которое нужно для загрузки офлайн-конверсий — точнее и уже, чем более широкое `metrika:write` (создание/изменение счётчиков), которое просить не нужно. Если в интерфейсе Яндекса формулировка чуть отличается от приведённой — не страшно, отметьте пункт, ближе всего описывающий «загрузка офлайн-данных/конверсий», и напишите мне точную формулировку из Шага 5.
6. Сохранить / создать приложение.
7. На открывшейся странице приложения скопировать и сохранить в надёжном месте:
   - **ID** — это `client_id`;
   - **Пароль** — это `client_secret`, показывается один раз. Если потеряется — на той же странице можно перевыпустить.

**В отличие от Директа:** у Метрики нет отдельной «заявки на доступ к API» с рассмотрением до 7 дней — регистрации приложения с нужным scope достаточно, доступ к самому API работает сразу.

## Шаг 2. Счётчик Метрики для теста

Нужен хотя бы один счётчик (тег) Метрики, на который у вашего Яндекс-аккаунта есть права редактирования — конверсии загружаются в конкретный `counterId`, и Метрика должна знать, что вы вправе в него писать.

- Свой действующий счётчик (если сайт уже подключён к Метрике) — идеально. `counterId` — число в настройках счётчика (Метрика → «Настройка» → «Номер счётчика», либо видно в URL страницы счётчика).
- Если действующего нет — создать тестовый: https://metrika.yandex.ru/ → «Добавить счётчик» → указать любой домен (даже ещё не привязанный физически — счётчик создастся и получит `counterId`, установка кода на сайт для этого research-таска не обязательна).

**Опционально, но полезно для Таска 3** (не обязательно для Таска 1): в этом счётчике создать одну простую цель («Цели» → «Добавить цель» → тип «JavaScript-событие» или «Составное событие»), задать понятный идентификатор, например `qualified_lead`. Если сделаете — пришлите идентификатор цели, это ускорит Таск 3.

## Шаг 3. Добавить в `.env`

В свой локальный `.env` (файл уже в `.gitignore`, наружу не уходит) добавить три переменные и заполнить первые две значениями из Шага 1:

```bash
# Яндекс Метрика OAuth (Phase 22.5, Task 1 — research)
YANDEX_METRIKA_OAUTH_CLIENT_ID=
YANDEX_METRIKA_OAUTH_CLIENT_SECRET=
YANDEX_METRIKA_OAUTH_REDIRECT_URI=http://localhost:3000/api/integrations/yandex/metrika/callback
```

Ничего в `.env.example` и `CLAUDE.md` сейчас добавлять не нужно — это часть Таска 2, когда появится код, который эти переменные читает.
