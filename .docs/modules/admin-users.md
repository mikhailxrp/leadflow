# Модуль: Управление пользователями (admin-users)

> Спецификация управления менеджерами администратором: создание, блокировка, разблокировка, удаление. Только для роли ADMIN.

## Цели модуля

После завершения этого модуля:

- Админ видит список всех пользователей компании
- Админ создаёт менеджеров (имя, email, пароль)
- Админ блокирует и разблокирует менеджеров
- Админ удаляет менеджеров (с подтверждением)
- Блокировка сохраняет историю действий менеджера
- Защита от потери доступа (нельзя удалить/заблокировать последнего админа и себя)

**Не входит в модуль:**

- Вход и сессии → `.docs/modules/auth.md`
- Регистрация — её нет (Boxed-модель); первый админ создаётся сидером
- Привязка Telegram менеджером → `.docs/modules/notifications.md`
- Влияние блокировки на round-robin → `.docs/modules/assignment.md`

## Архитектурные решения

### 1. Админ и менеджеры — одна таблица `User`, поле `role`

В отличие от изолированных таблиц в Boxed-проекте без SaaS, у нас все пользователи компании в `User`, различаются `role` (`ADMIN | MANAGER`). Причина — тенант владеет всеми своими пользователями (см. `.docs/database.md` → модель User, `CLAUDE.md` → мультитенантность).

### 2. Блокировка вместо удаления — основной путь

`User.isBlocked` — мягкое отключение. Заблокированный менеджер не входит (проверка в `authorize()`), но его комментарии, назначенные лиды и история сохраняются. Удаление — крайняя мера, необратимо.

```
Рекомендация UI: блокировка на видном месте, удаление — за подтверждением.
Блокировка обратима, удаление — нет.
```

### 3. Пароль задаёт админ при создании

Plain-пароль вводит админ в форме создания, сервер хэширует (bcrypt) и сохраняет только хэш. Email-доставки пароля нет (out of scope MVP) — админ сообщает пароль менеджеру сам. Смена пароля менеджеру — повторным редактированием админом.

## Список пользователей

Страница `/admin/users`. Только ADMIN. Таблица (`components/admin/users/UsersTable.tsx`).

Колонки: Пользователь (аватар-инициалы + имя + email), Роль, Статус (Активен / Заблокирован), Дата создания, Действия.

- Роль — бейдж: «Администратор» (акцентный) / «Менеджер» (нейтральный)
- Статус — бейдж: «Активен» (зелёный dot) / «Заблокирован» (красный dot)
- Действия — иконки: редактировать, блокировать/разблокировать, удалить

Все пользователи — только своей компании (`where: { companyId }`).

## Создание менеджера

### Поведение (FR-53)

Кнопка «Добавить менеджера» → модалка (`components/admin/users/AddUserModal.tsx`): Имя, Email, Пароль.

`POST /api/users`:

```typescript
// сервер
const email = body.email.toLowerCase().trim();
// проверка уникальности email
const exists = await prisma.user.findUnique({ where: { email } });
if (exists) return error("EMAIL_EXISTS");

const passwordHash = await hashPassword(body.password);  // bcrypt
const user = await prisma.user.create({
  data: { companyId: session.companyId, email, name, passwordHash, role: "MANAGER" },
});
await writeEvent(session.companyId, "USER_CREATED", { userId: user.id }, session.user.id);
```

Создаются только менеджеры (`role = MANAGER`). Создание других админов — задел; в MVP админ один (из сидера). Если нужно — добавляется как право (вне базового scope).

## Блокировка и разблокировка

### Поведение (FR-54)

`PATCH /api/users/:id` с `{ isBlocked: true | false }`.

```
Блокировка (isBlocked = true):
  → менеджер не сможет войти (проверка в authorize())
  → активная сессия живёт до истечения (JWT) — см. примечание ниже
  → исключается из round-robin (assignment)
  → его лиды и комментарии сохраняются
  → событие USER_BLOCKED

Разблокировка (isBlocked = false):
  → менеджер снова может войти
  → снова участвует в round-robin
  → событие USER_UNBLOCKED
```

### Примечание про активную сессию

JWT-сессия живёт до `maxAge` (24ч). Заблокированный менеджер с активной сессией технически сохраняет доступ до её истечения. Для MVP это приемлемо (24ч). Если нужна немедленная блокировка — проверять `isBlocked` в `proxy.ts`/на чувствительных эндпоинтах (задел, не в базовом MVP).

## Удаление менеджера

### Поведение

`DELETE /api/users/:id`. UI — модалка подтверждения «Удалить пользователя? Действие необратимо».

```
При удалении:
  → User удаляется
  → его Comment остаются (userId без каскада — история обсуждений не теряется)
  → его назначенные лиды: assignedToId становится null (лиды нераспределены)
  → событие USER_DELETED (в журнал компании, payload: снимок имени/email)
```

**Почему комментарии остаются:** удаление менеджера не должно стирать историю работы с лидами. Поэтому связь `Comment.userId` — без каскадного удаления (см. `.docs/database.md` → модель Comment). В UI у «осиротевшего» комментария показывается «Удалённый пользователь».

## Защитные инварианты

Критичные проверки на сервере — защита от потери доступа к системе.

```
1. Нельзя удалить самого себя
   → admin не может удалить запись со своим id из сессии
   → иначе можно случайно выпилить свой доступ

2. Нельзя заблокировать самого себя
   → тот же риск — потеря доступа

3. Нельзя удалить последнего администратора
   → если в компании останется 0 админов — системой никто не управляет
   → проверка: count(role=ADMIN, companyId) > 1 перед удалением админа

4. Нельзя понизить роль последнего администратора
   → тот же инвариант (если редактирование роли вообще доступно)

5. Email уникален глобально
   → проверка перед созданием (с учётом lowercase+trim)
```

Эти проверки — **на сервере**, не на клиенте. Клиентские можно обойти.

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
| --- | --- | --- | --- | --- |
| GET | `/api/users` | Список пользователей компании | Session | ADMIN only |
| POST | `/api/users` | Создать менеджера | Session | ADMIN only |
| PATCH | `/api/users/:id` | Блокировать / разблокировать / изменить | Session | ADMIN only |
| DELETE | `/api/users/:id` | Удалить менеджера | Session | ADMIN only |

### `POST /api/users`

**Request:** `{ "name": "Алексей", "email": "alex@company.ru", "password": "..." }`
**Response 200:** `{ "id": "clxxx", "name": "Алексей", "email": "alex@company.ru", "role": "MANAGER", "isBlocked": false }`
**Response 400:** `{ "success": false, "error": "EMAIL_EXISTS" | "VALIDATION_ERROR" }`

### `PATCH /api/users/:id`

**Request (блокировка):** `{ "isBlocked": true }`
**Request (изменить данные):** `{ "name": "Алексей П.", "password": "новый" }`
**Response 200:** `{ "success": true }`
**Response 400:** `{ "success": false, "error": "CANNOT_BLOCK_SELF" }`

### `DELETE /api/users/:id`

**Response 200:** `{ "success": true }`
**Response 400:**

```json
{ "success": false, "error": "CANNOT_DELETE_SELF" | "CANNOT_DELETE_LAST_ADMIN" }
```

## Серверные правила безопасности

1. **Все операции — только ADMIN.** Проверка роли на сервере в каждом эндпоинте. Менеджер не управляет пользователями.

2. **Все запросы — с `companyId` из сессии.** Админ видит и меняет только пользователей своей компании. В SaaS-будущем — не дотянется до чужого тенанта.

3. **Защитные инварианты на сервере:** нельзя удалить/заблокировать себя, нельзя удалить последнего админа. Проверки до выполнения операции.

4. **Email — lowercase + trim** перед проверкой уникальности и записью. Иначе дубли/коллизии.

5. **Пароль хэшируется (bcrypt) перед записью.** Plain нигде не хранится и не возвращается. В ответах User-объектов нет `passwordHash`.

6. **`role` нельзя задать произвольно через API.** Создание — всегда `MANAGER`. Повышение до ADMIN — отдельное контролируемое действие (вне базового MVP).

7. **`companyId` не берётся из тела запроса** — только из сессии. Иначе админ мог бы создать пользователя в чужой компании.

8. **События USER_* пишет сервер** через `lib/events.ts` — создание, блокировка, разблокировка, удаление. Аудит действий админа.
