## Цели модуля

После завершения этого модуля:

- Админ видит инструкции и URL вебхуков для Tilda и WordPress
- Админ генерирует API-ключи для сторонних сайтов
- Ключ показывается один раз при создании, дальше — только хэш
- Админ удаляет ключи
- Админ переключает режим интеграции Яндекс Директа (полный / UTM-only)
- Статусы подключений видны в UI

**Не входит в модуль:**

- Сам приём лидов по ключам/вебхукам → `.docs/modules/leads-intake.md`
- Логика парсинга полей → `.docs/modules/leads-intake.md`
- Переключатели распределения/видимости → `.docs/modules/app-settings.md`

## Архитектурные решения

### 1. Это UI-обвязка над приёмом

Модуль не принимает лиды сам — он управляет тем, **чем** аутентифицируется приём (API-ключи) и **как** настроен (режим Яндекса). Сам приём — в `leads-intake`.

### 2. Ключ — как пароль: показать один раз, хранить хэш

При создании генерируется случайный ключ, показывается админу один раз с кнопкой «Скопировать». В БД — только `keyHash` (см. `.docs/database.md` → модель ApiKey). Потерян → создать новый, старый удалить.

### 3. Режим Яндекса — флаг в настройках

Полный/минимальный режим Директа (`.docs/modules/leads-intake.md`) переключается флагом. Где хранить — в `Company.settings` (рядом с остальными настройками) или отдельным полем; решает реализация. По умолчанию — минимальный (UTM-only), он всегда работает.

## Tilda — инструкция и вебхук

Блок-карточка на странице интеграций (`design-system.md` → настройки интеграций, карточка 1).

Содержимое:

- Иконка + «Tilda» + статус подключения (активно / не настроено)
- Инструкция: «В настройках формы Tilda → Webhook укажите адрес:»
- Поле с URL `https://<домен>/api/webhooks/tilda` + кнопка «Скопировать»
- Подсказка: Tilda при сохранении вебхука шлёт тестовый запрос — система ответит OK (см. `leads-intake` → обработка test)

Статус «активно» определяется фактом наличия принятых лидов с `source = TILDA` (или просто статичная инструкция в MVP — уточняет реализация).

## Универсальный webhook — API-ключи

Блок-карточка (карточка 3) — основная функциональная часть модуля.

### Список ключей

Таблица: Название, Источник (`sourceLabel`), Ключ (скрыт `••••••••`), Дата создания, Действия (удалить).

### Создание ключа

Кнопка «Создать ключ» → модалка: Название (для какого сайта), Источник (метка `source`).

```
POST /api/api-keys { name, sourceLabel }
  → сервер генерирует случайный ключ (напр. "sk_live_" + 32 случайных символа)
  → хэширует, сохраняет ApiKey { companyId, name, keyHash, sourceLabel }
  → возвращает PLAIN-ключ ОДИН РАЗ
  → UI показывает ключ + кнопку «Скопировать» + предупреждение «больше не покажем»
```

### Использование ключа

Админ вставляет ключ в URL вебхука на своём сайте:
`https://<домен>/api/webhooks/leads?key=<КЛЮЧ>&source=<любой_source>`

Приём по этому ключу — в `leads-intake`.

### Удаление ключа

`DELETE /api/api-keys/:id` — с подтверждением. После удаления вебхуки с этим ключом перестают приниматься (401).

## Яндекс Директ — режим интеграции

Блок-карточка (карточка 4).

### Переключатель режима

```
Режим «Минимальный (UTM)» [по умолчанию]
  → лиды с Яндекса принимаются по UTM-меткам, как обычные лиды с лендинга
  → ничего настраивать не нужно, работает сразу

Режим «Полный (API)»
  → требует OAuth-доступа к рекламному кабинету Яндекс Директа
  → даёт campaign/adGroup/keyword/device/region
  → кнопка «Подключить кабинет» (OAuth-флоу)
```

### Поведение UI

- Селектор/переключатель режима
- Для полного режима — статус OAuth-подключения и кнопка подключения
- Подсказка: «Минимальный режим работает всегда. Полный требует доступа к кабинету Яндекса.»

Детали приёма в каждом режиме — `.docs/modules/leads-intake.md` → «Источник: Яндекс Директ». OAuth-флоу к Яндексу — отдельная подзадача фазы (может быть отложен, если доступ к кабинету не получен к старту — минимальный режим закрывает MVP).

## Генерация и хранение ключей

```typescript
// lib/intake/apiKey.ts — концепт
import { randomBytes, createHash } from "crypto";

function generateApiKey(): string {
  return "sk_live_" + randomBytes(24).toString("base64url"); // ~32 символа
}

function hashKey(plain: string): string {
  return createHash("sha256").update(plain).digest("hex");
}

// при создании:
const plain = generateApiKey();
const keyHash = hashKey(plain);
await prisma.apiKey.create({ data: { companyId, name, keyHash, sourceLabel } });
return plain; // показать ОДИН раз

// при приёме (leads-intake):
const incoming = req.query.key;
const key = await prisma.apiKey.findFirst({ where: { keyHash: hashKey(incoming) } });
// null → 401
```

Тот же `hashKey` используется при создании и при проверке. Plain-ключ нигде не хранится.

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
| --- | --- | --- | --- | --- |
| GET | `/api/api-keys` | Список ключей (без plain) | Session | ADMIN only |
| POST | `/api/api-keys` | Создать ключ (вернуть plain один раз) | Session | ADMIN only |
| DELETE | `/api/api-keys/:id` | Удалить ключ | Session | ADMIN only |
| PATCH | `/api/settings` | Сменить режим Яндекса (общий settings-эндпоинт) | Session | ADMIN only |

> Режим Яндекса меняется через общий `PATCH /api/settings` (см. `.docs/modules/app-settings.md`), отдельного эндпоинта не заводим.

### `POST /api/api-keys`

**Request:** `{ "name": "Лендинг весенней акции", "sourceLabel": "landing-spring" }`
**Response 200 (ключ показывается единожды):**

```json
{ "id": "clxxx", "name": "Лендинг весенней акции", "sourceLabel": "landing-spring",
  "key": "sk_live_AbC123...", "warning": "Сохраните ключ — он больше не будет показан" }
```

### `GET /api/api-keys`

**Response 200 (без plain-ключа):**

```json
{ "keys": [
  { "id": "clxxx", "name": "Лендинг весенней акции", "sourceLabel": "landing-spring",
    "createdAt": "2026-06-07T..." }
] }
```

### `DELETE /api/api-keys/:id`

**Response 200:** `{ "success": true }`

## Серверные правила безопасности

1. **Все операции — только ADMIN.** Управление интеграциями — админское право. Проверка роли на сервере.

2. **Все запросы — с `companyId` из сессии.** Ключи изолированы по компании. Админ не видит/не удаляет чужие ключи.

3. **Plain-ключ возвращается ровно один раз** — при создании. `GET /api/api-keys` никогда не отдаёт plain или хэш, только метаданные.

4. **Ключ хранится хэшированным** (`sha256`). Тот же алгоритм при проверке в приёме.

5. **Генерация — криптостойкая** (`crypto.randomBytes`), не `Math.random()`. Ключ должен быть непредсказуем.

6. **Удаление ключа немедленно прекращает приём** по нему (приём ищет по `keyHash`, записи больше нет → 401).

7. **`companyId` и `sourceLabel`** не берутся из тела приёмного запроса — `companyId` из найденного ключа, источник из ключа/параметра. Сторонний сайт не подменит компанию.

8. **OAuth-токены Яндекса** (если полный режим) хранятся на сервере, не на клиенте. В `.env` или зашифрованном поле — решает реализация. Никогда не отдаются клиенту.
