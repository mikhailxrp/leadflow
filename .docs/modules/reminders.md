# Модуль: Напоминания (reminders)

> Спецификация создания, управления и доставки напоминаний по лидам. Изменения относительно предыдущей версии минимальны — большая часть механики не затронута синхронизацией с Техничкой.
> Связанные файлы: `.docs/database.md` (`Reminder`, `ReminderStatus`), `.docs/modules/notifications.md` (Telegram-доставка), `.docs/modules/leads.md`.

---

## Что не изменилось

Всё ядро модуля — без изменений относительно предыдущей версии:

- Каналы в JSONB (`Reminder.channels`), не enum
- Статус `FIRED` выставляется до реальной доставки, ошибки — в `events` как `REMINDER_FAILED`
- Idempotent guard через `updateMany({ where: { status: "PENDING" } })`
- Cron `node-cron`, раз в минуту
- Напоминание привязано к лиду, `createdBy` — для отображения; удаление/блокировка пользователя не останавливает доставку
- CRUD-эндпоинты, валидация (`remindAt` в будущем, минимум один канал), отображение в карточке (активные/история)

Полные детали — см. предыдущую версию документа, она остаётся источником правды по этим разделам.

---

## Что изменилось

### Создание/изменение/отмена — не зависит от блокировки компании на уровне отдельного guard'а

Создание, редактирование и отмена напоминания — обычные мутации. Отдельной проверки подписки/тарифа не существует — биллинга в продукте нет. Единственная точка проверки доступа компании — `Company.isBlocked` при входе (`.docs/modules/auth.md`), не на уровне отдельных эндпоинтов.

```
Заблокированная компания:
  Войти                        → нельзя (проверка в authorize())
  Уже запланированное PENDING  → срабатывает по расписанию как обычно (это не действие пользователя)
```

Уже созданные `PENDING`-напоминания **продолжают срабатывать** даже у заблокированной компании — тот же принцип «лид нельзя потерять», применённый к уже взятым на себя обязательствам: остановка запланированной доставки была бы неожиданным побочным эффектом блокировки, который создаёт риск пропущенной договорённости с клиентом.
