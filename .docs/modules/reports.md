# Модуль: Отчёты (reports)

> Спецификация минимальной аналитики: лиды по периоду, скорость ответа, конверсия, причины отказа, по сотрудникам, по источникам. Единый набор для всех компаний — деления по тарифу больше не существует. Построен полностью на существующих данных (`Lead`/`Event`/`Task`), без новых таблиц.
> Связанные файлы: `.docs/database.md`, `.docs/modules/leads.md` (переиспользуемый счётчик менеджера), `.docs/modules/risk.md`.

## Цели модуля

После завершения этого модуля:

- Руководитель и Администратор видят: лиды по периоду, скорость первого ответа, конверсия по этапам, причины отказа, разбивку по сотрудникам и источникам
- Доступен экспорт в CSV
- Все цифры — производные от `events`/`Lead`/`Task`, без отдельной таблицы метрик и без дублирования логики, уже написанной для других экранов

**Не входит в модуль:**

- Доступ Менеджера к отчётам — отчёты остаются возможностью Руководителя и Администратора (раздел 3 `prd.md`), не Менеджера. Экран защищён как `(management)`, не как обычная рабочая зона (см. `CLAUDE.md`)
- Деление глубины отчётов по тарифу — тарифов не существует, набор один для всех компаний
- Сквозная аналитика, ROI, прогноз выручки — вне scope (см. `prd.md` → «Вне scope»)
- Счётчик активности менеджеров как управленческий инструмент реального времени — это `/control` (`.docs/modules/leads.md`); здесь — тот же расчёт, но в составе общего отчёта за произвольный период, не отдельная логика

## Архитектурные решения

### Переиспользование, не дублирование

«По сотрудникам» в этом модуле — это `lib/control/getManagerStats.ts` (уже написан для `/control`, см. `.docs/modules/leads.md`), вызванный с другим диапазоном периода (`periodStart`/`periodEnd` вместо только «последние N дней») и в другом месте интерфейса. Площадка другая, расчёт — один и тот же. Дублировать формулы под новым именем — риск, что они разойдутся при будущих правках.

### Доступ — `requireCompanyAccess`, не прямая проверка роли

Отчёты доступны Руководителю, Администратору **и маркетологу** в режиме входа внутрь компании (allow-list, `constants/marketerAccess.ts`). У маркетолога нет `session.user` (только `session.marketer`) — прямая проверка `session.user.role` бросила бы исключение на его сессии. Поэтому каждый route handler использует единый guard, учитывающий оба типа company-актора:

```typescript
// GET /api/reports/* — каждый route handler
const actor = await requireCompanyAccess({
  minRole: "HEAD",
  method: "GET",
  pathname: "/api/reports/summary", // свой путь для каждого эндпоинта
});
// actor.companyId — для user и для marketer одинаково доступен
```

Раньше здесь была проверка тарифа (`requirePlan`) — убрана полностью вместе с биллингом.

## Состав отчётов

### Лиды по периоду

```
Сколько лидов поступило за период (группировка по дню/неделе для графика)
```

### Скорость первого ответа

```
Среднее время от LEAD_CREATED до LEAD_TAKEN_IN_WORK, за период
(лиды без LEAD_TAKEN_IN_WORK в периоде — не входят в среднее, но считаются отдельно как "необработанные")
```

### Конверсия по этапам

```
Для каждого этапа: сколько лидов сейчас в нём И сколько лидов когда-либо через него проходило (по STAGE_CHANGED)
Конверсия в финальную сделку = count(closeType=WON) / count(всех лидов за период)
```

Сколько осталось необработанных, сколько зависших, сколько без следующего действия (FR-150-2) — переиспользование `lib/risk/computeRiskBatch.ts` (`.docs/modules/risk.md`), агрегированное по периоду.

### Причины отказа

```
group by lossReasonId, count(Lead) where closeType = LOST, за период
```

### По сотрудникам

Переиспользует `getManagerStats()` — за произвольный период, не только «текущий месяц», как в `/control`.

### По источникам

```
group by source, count(Lead), conversion (closeType=WON / total), за период
```

### Экспорт CSV

Любой из вышеперечисленных отчётов — кнопка «Экспорт» → CSV с теми же данными, что на экране. Доступен всем, у кого есть доступ к самому отчёту — отдельного ограничения на экспорт не существует.

## Соотношение со страницей «Контроль»

| | `/control` | `/reports` |
| --- | --- | --- |
| Назначение | Операционное управление здесь и сейчас | Аналитика за период |
| Период | Текущий месяц по умолчанию | Произвольный, выбираемый |
| Доступ | Руководитель, Администратор | Руководитель, Администратор, маркетолог (allow-list) |
| Расчёт «по сотрудникам» | `getManagerStats()` | Тот же `getManagerStats()`, другой период/UI |

Это не два конкурирующих модуля — `reports.md` использует часть того же кода, что и `leads.md` → «Страница контроля», под другим экраном с другим назначением. Оба теперь живут в зоне `(management)`, не `(admin)` (см. `CLAUDE.md`).

## Серверные правила безопасности

1. **Все эндпоинты `/api/reports/*` — `requireCompanyAccess({ minRole: "HEAD", method, pathname })`.** Для `user`-актора это `hasMinRole(role, "HEAD")`; для `marketer`-актора — сверка allow-list по пути и методу (`constants/marketerAccess.ts`). Менеджер не имеет доступа к отчётам ни в каком виде — ни как `user` (роль ниже HEAD), ни как маркетолог (allow-list не про роль компании), проверяется на сервере, не только скрытием пункта меню.
2. **`companyId` из актора** (`actor.companyId`, одинаково для `user` и `marketer`) — отчёты не пересекают тенантов.
3. **Экспорт — без чувствительных данных сверх того, что и так видно в отчёте** (без хэшей, токенов, внутренних идентификаторов сверх необходимого).
