## Цели модуля

После завершения этого модуля:

- Лиды отображаются на Kanban-доске по этапам-колонкам
- Карточку лида можно перетащить между этапами (drag-and-drop)
- Дата смены статуса фиксируется автоматически
- Смена этапа пишется в историю лида
- Админ создаёт, переименовывает, переупорядочивает и удаляет этапы
- Удаление этапа с лидами требует переноса лидов в другой этап
- Воронка полностью настраиваема, не хардкод

**Не входит в модуль:**

- Приём лидов и присвоение начального этапа → `.docs/modules/leads-intake.md`
- Список лидов, карточка, поиск → `.docs/modules/leads.md`
- Назначение менеджера → `.docs/modules/assignment.md`
- 5 дефолтных этапов создаёт сидер → `.docs/database.md` → «Сидеры»

## Архитектурные решения

### 1. Этапы — данные, не код

Этапы хранятся в таблице `PipelineStage` (поля `companyId`, `name`, `color`, `order`). Никакого хардкода названий или количества. Админ управляет ими через UI (FR-22). У каждой компании своя воронка (мультитенантность).

### 2. Текущий этап лида — FK, не enum

`Lead.stageId → PipelineStage.id`. Не enum-поле, потому что набор этапов динамический. Удаление/добавление этапа не требует миграции.

### 3. Порядок через целое поле `order`

Колонки сортируются по `PipelineStage.order` (0, 1, 2…). Переупорядочивание = пересчёт `order` у затронутых этапов в одной транзакции.

### 4. Дата смены статуса — из событий, не из колонки

Отдельного поля «дата последней смены» в `Lead` нет. Дата = `createdAt` события `STAGE_CHANGED`. История перемещений (FR-26) — выборка таких событий по лиду. Это согласуется с Event Sourcing (`.docs/database.md`).

### 5. Drag-and-drop через @dnd-kit

Библиотека `@dnd-kit/core` (см. `CLAUDE.md` → стек). Оптимистичное обновление UI: карточка переезжает сразу, запрос уходит фоном; при ошибке — откат на исходную колонку.

## Kanban-доска

### Отображение

Страница `/pipeline`. Компонент `components/pipeline/KanbanBoard.tsx`.

- Колонки = этапы, отсортированы по `order`
- Каждая колонка: цветная полоска сверху (`stage.color`), название, счётчик лидов
- Карточка лида в колонке: имя, телефон, бейдж источника, ответственный менеджер (см. `design-system.md` → карточка лида в Kanban)
- Горизонтальный скролл если этапов много
- На мобильном — вертикальный стек колонок (`design-system.md` → адаптивность)

### Источник данных

Лиды для доски выбираются с учётом видимости (`visibilityWhere` из `.docs/modules/leads.md`): менеджер в режиме `OWN` видит на доске только свои карточки.

Для производительности доска грузит лиды порциями (например, по N последних на колонку) с кнопкой «показать ещё», а не все лиды компании сразу. Точную стратегию уточняет реализация фазы.

### Клик по карточке

Открывает карточку лида (`.docs/modules/leads.md`).

## Drag-and-drop карточек

### Поведение (FR-21)

Перетаскивание карточки из одной колонки в другую меняет `Lead.stageId`.

```
Пользователь тянет карточку из «Новый лид» в «В работе»
  → UI: карточка сразу переезжает (оптимистично)
  → PATCH /api/leads/:id { stageId: newStageId }
  → сервер: транзакция (update stageId + event STAGE_CHANGED)
  → успех: ничего не меняем (UI уже верный)
  → ошибка: откат карточки в исходную колонку + toast об ошибке
```

### Транзакция смены этапа (FR-24)

Полный код — `.docs/database.md` → «Критичные транзакции → Смена этапа». Кратко:

```typescript
await prisma.$transaction(async (tx) => {
  const lead = await tx.lead.findFirstOrThrow({ where: { id, companyId } }); // изоляция + текущий этап
  await tx.lead.update({ where: { id }, data: { stageId: newStageId } });
  await tx.event.create({
    data: { companyId, leadId: id, userId, type: "STAGE_CHANGED",
            payload: { fromStageId: lead.stageId, toStageId: newStageId } },
  });
});
```

Дата смены = `createdAt` созданного события. Отдельное поле не нужно.

### Кто может двигать

- ADMIN — любые карточки
- MANAGER — карточки в пределах своей видимости (`ALL` или `OWN`)

Смена статуса разрешена менеджеру (FR — роли: «Изменение статуса лида»).

## Настройка этапов (админ)

Страница `/admin/pipeline-settings`. Только ADMIN. Список этапов с drag-and-drop переупорядочиванием (см. `design-system.md` → настройки воронки).

### Создание (FR-22)

`POST /api/stages` — `{ name, color }`. Новый этап добавляется в конец (`order = max(order) + 1`). Цвет из палитры или произвольный hex.

### Переименование (FR-22)

`PATCH /api/stages/:id` — `{ name }` и/или `{ color }`. Inline-редактирование в списке.

### Удаление — см. отдельный раздел ниже (требует переноса лидов).

### Минимум этапов

Нельзя удалить последний этап — воронка не может быть пустой (лидам некуда попадать при приёме). Проверка на сервере: если этапов останется 0 — `400`.

## Удаление этапа с переносом лидов

### Проблема

Этап может содержать лиды. Просто удалить — значит осиротить лиды (их `stageId` укажет на несуществующий этап). Поэтому удаление этапа с лидами требует указать, **куда перенести** лиды (FR-25).

### Поведение

```
Если → в удаляемом этапе 0 лидов
То   → удаляем сразу (DELETE /api/stages/:id)

Если → в удаляемом этапе N > 0 лидов
То   → UI показывает модалку «В этапе N лидов. Куда перенести?»
       → админ выбирает целевой этап
       → DELETE /api/stages/:id?moveTo=<targetStageId>
       → транзакция: перенос лидов + удаление этапа
```

### Транзакция

Полный код — `.docs/database.md` → «Критичные транзакции → Удаление этапа с переносом лидов». Кратко:

```typescript
await prisma.$transaction(async (tx) => {
  // 1. Перенести все лиды в целевой этап
  const moved = await tx.lead.updateMany({
    where: { companyId, stageId: stageToDelete },
    data: { stageId: targetStage },
  });
  // 2. Событие о массовом переносе
  await tx.event.create({
    data: { companyId, userId: adminId, type: "STAGE_CHANGED",
            payload: { fromStageId: stageToDelete, toStageId: targetStage, bulk: moved.count } },
  });
  // 3. Удалить пустой этап
  await tx.pipelineStage.delete({ where: { id: stageToDelete } });
});
```

### Защиты

- `moveTo` обязателен, если в этапе есть лиды (иначе `400`)
- `moveTo` не может равняться удаляемому этапу
- Целевой этап должен принадлежать той же компании
- Нельзя удалить последний этап компании

## Переупорядочивание этапов

### Поведение

Админ перетаскивает этапы в списке настроек (drag-and-drop через @dnd-kit). После перестановки — кнопка «Сохранить порядок» или автосохранение.

`PATCH /api/stages/reorder` — `{ orderedIds: [id1, id2, id3, ...] }`. Сервер пересчитывает `order` по позиции в массиве, в одной транзакции.

```typescript
await prisma.$transaction(
  orderedIds.map((id, index) =>
    prisma.pipelineStage.update({
      where: { id, companyId },     // изоляция тенанта
      data: { order: index },
    })
  )
);
```

Новый порядок сразу отражается на Kanban-доске (колонки сортируются по `order`).

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
| --- | --- | --- | --- | --- |
| GET | `/api/stages` | Список этапов компании | Session | ADMIN / MANAGER |
| POST | `/api/stages` | Создать этап | Session | ADMIN only |
| PATCH | `/api/stages/:id` | Переименовать / сменить цвет | Session | ADMIN only |
| PATCH | `/api/stages/reorder` | Переупорядочить | Session | ADMIN only |
| DELETE | `/api/stages/:id` | Удалить (с переносом лидов) | Session | ADMIN only |
| PATCH | `/api/leads/:id` | Сменить этап лида (drag-and-drop) | Session | Видимость по роли |

> Смена этапа лида использует общий `PATCH /api/leads/:id` (см. `.docs/modules/leads.md`), передавая `stageId`. Отдельного эндпоинта не заводим.

### `POST /api/stages`

**Request:** `{ "name": "Согласование", "color": "#6366F1" }`
**Response 200:** `{ "id": "clxxx", "name": "Согласование", "color": "#6366F1", "order": 5 }`

### `DELETE /api/stages/:id`

**Request:** `DELETE /api/stages/clxxx?moveTo=clyyy`
**Response 200:** `{ "success": true, "moved": 31 }`
**Response 400:**

```json
{ "success": false, "error": "MOVE_TARGET_REQUIRED" | "CANNOT_DELETE_LAST_STAGE" | "INVALID_MOVE_TARGET" }
```

### `PATCH /api/stages/reorder`

**Request:** `{ "orderedIds": ["clA", "clB", "clC"] }`
**Response 200:** `{ "success": true }`

## Серверные правила безопасности

1. **Все запросы этапов — с фильтром `companyId`.** Этапы изолированы по тенанту. Нельзя получить/изменить этап чужой компании.

2. **Создание, переименование, удаление, переупорядочивание — только ADMIN.** Менеджер может лишь двигать карточки лидов (менять `stageId`), но не управлять самими этапами.

3. **Смена `stageId` лида проверяет видимость** (`visibilityWhere`) — менеджер в режиме `OWN` не подвинет чужой лид.

4. **Целевой этап при удалении — той же компании.** Проверка `targetStage.companyId === session.companyId`. Иначе лиды «утекут» в чужую воронку (в SaaS-будущем).

5. **Нельзя удалить последний этап** — приёму лидов нужен хотя бы один этап (стартовый). Проверка перед удалением.

6. **`moveTo` обязателен при непустом этапе.** Сервер сам проверяет наличие лидов, не доверяет клиенту. Если лиды есть, а `moveTo` не передан — `400`.

7. **Переупорядочивание — в транзакции.** Частичный пересчёт `order` (если упадёт на середине) оставит воронку в несогласованном виде. Только всё или ничего.

8. **Смена этапа пишет событие** `STAGE_CHANGED` через `lib/events.ts`. Это источник истории и даты смены статуса.
