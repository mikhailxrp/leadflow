# Модуль: Импорт CSV/Excel (import)

> Спецификация импорта лидов из Excel/CSV: загрузка, маппинг колонок, превью с дублями, батч-создание, отчёт, откат. Новый модуль.
> Связанные файлы: `.docs/database.md` (`ImportBatch`, `Lead.importBatchId`), `.docs/modules/leads-intake.md` (переиспользуемая логика дублей и создания лида), `.docs/modules/leads.md`.

## Цели модуля

После завершения этого модуля:

- Пользователь загружает Excel/CSV, видит колонки файла и сопоставляет их с полями Лид-Канал
- Перед подтверждением виден список возможных дублей
- После подтверждения — отчёт: сколько загружено, пропущено, дублей, ошибок
- Источник импортированных лидов — `"import"`
- Можно отменить последний импорт целиком

**Не входит в модуль:**

- Прямая интеграция с Google Таблицами — на этом этапе только экспорт в CSV и инструкция (FR-140-7)
- Импорт других сущностей (задачи, комментарии) — только лиды

## Архитектурные решения

### 1. Файл не сохраняется — обрабатывается транзитом

Никакого файлового хранилища (S3 и т.п.) не вводится. Файл парсится на сервере в рамках одного запроса (`xlsx`/`papaparse`), результат — массив строк в JSON — возвращается клиенту для маппинга и превью. Подтверждение присылает обратно уже размеченные данные, не файл повторно. Исходный файл нигде не сохраняется ни до, ни после импорта.

**Почему так:** хранить файл нужно было бы только для случая «показать его ещё раз позже» — этого сценария нет (есть `ImportBatch`-отчёт и возможность отката, которые не требуют исходного файла). Не усложняем инфраструктуру ради сценария, которого нет.

### 2. Дедуп переиспользует существующую логику

Превью дублей в импорте — это та же функция, что и при ручном создании лида (`.docs/modules/leads-intake.md` → `flagPossibleDuplicates`), вызванная в режиме «только проверка», без записи — как уже устроено для ручного создания одного лида (`.docs/modules/leads.md`).

### 3. Батч — атомарная единица отмены

`ImportBatch` — не просто отчёт, а ключ для отмены: «откатить импорт» = удалить все лиды с этим `importBatchId` одной транзакцией. Подробности — `.docs/database.md` → «Критичные транзакции → Откат импорта».

## Флоу импорта

```
1. Загрузка файла → POST /api/import/preview (multipart, файл в теле запроса)
2. Сервер парсит → возвращает: список колонок файла + первые N строк + предложенный авто-маппинг
3. Пользователь подтверждает/правит маппинг колонок → поля Лид-Канал
4. Клиент отправляет уже размеченные строки на повторную проверку →
   POST /api/import/preview (тот же эндпоинт, второй вызов — теперь с маппингом, не с файлом)
5. Сервер прогоняет дедуп-проверку по всем строкам → возвращает превью с пометками "возможный дубль"
6. Пользователь подтверждает → POST /api/import/confirm
7. Сервер: транзакция batch-создания (см. ниже) → отчёт
```

Шаги 2 и 4–5 идут через один и тот же эндпоинт с разными стадиями данных в теле — отдельного «эндпоинта на каждый шаг визарда» не заводим.

## Маппинг колонок

### Авто-предложение

Сервер пытается сопоставить колонки файла известным полям по тем же алиасам, что используются при приёме лидов (`constants/fieldAliases.ts`, переиспользуется, не дублируется).

```
Колонка файла "Имя клиента" → name (по алиасу)
Колонка файла "Тел."        → phone (по алиасу)
Колонка файла "Откуда"       → не распознана → пользователь сопоставляет вручную или оставляет как custom field
```

### Ручная корректировка

UI — таблица: слева колонки файла с примером значения, справа — селект поля Лид-Канал (включая «Не импортировать» и «Дополнительное поле» — попадёт в `customFields` под именем колонки).

---

## Превью и дубли

После маппинга — таблица предпросмотра (например, первые 50 строк из файла + агрегированные счётчики по всему файлу):

```
Будет создано: 312 лидов
Возможных дублей: 18 (отмечены, не блокируют — будут созданы с DuplicateFlag, как и при обычном приёме)
Ошибок в строках: 3 (например, полностью пустая строка) — будут пропущены, не остановят весь импорт
```

**Важно:** дубли в импорте **не блокируют создание**, как и везде в продукте — только помечаются. Решение «не сливать» (раздел 4.2 `prd.md`) одинаково для всех путей создания лида.

## Подтверждение и батч-создание

### Транзакция

Реализация — `lib/import/runImport.ts` (не буквально код из `.docs/database.md` — уточнено при планировании Phase 20, таск 2). Создаётся `ImportBatch` (`status: PROCESSING`). Строки обрабатываются **строго последовательно** (не параллельно) — так внутрифайловый дедуп видит уже закоммиченные более ранние строки того же файла. Для каждой строки: маппинг → полностью пустая после маппинга строка → `errors++`, пропуск; иначе — **синхронный** дедуп (`findPossibleDuplicates`, не «асинхронно, как при вебхук-приёме» — на этом шаге нет внешнего источника, ждущего `200 OK`, поэтому синхронность уместна и делает `ImportBatch.duplicates` точным) → per-row транзакция: лид (`source: "import"`, `importBatchId`, `assignedToId: null`) + событие `LEAD_CREATED` + по каждому совпадению `DuplicateFlag` + событие `DUPLICATE_FLAGGED`. Непредвиденное исключение на строке → `skipped++`, батч продолжается. По завершении — `ImportBatch.status = DONE`, агрегаты (`imported`/`skipped`/`duplicates`/`errors`) проставлены, событие `IMPORT_COMPLETED`.

**Не используется `lib/intake/createLead.ts`/`normalizeLead()`** — `createLead()` не принимает `importBatchId` и жёстко ставит `assignedToId: null` без параметра, что сделало бы откат (удаление по `importBatchId`) неработоспособным; `normalizeLead()` заново прогнал бы алиасы по уже размеченным ключам и не понимает `customField:<имя>`. `runImport.ts` строит лид из `mapping` самостоятельно, переиспользуя только дедуп-хелпер (`findPossibleDuplicates`) и раскладку строки (`lib/import/mapRow.ts`, общая с превью).

### Назначение и уведомления при импорте

Импортированные лиды **не уведомляются и не назначаются автоматически** (ни SSE-тост, ни Telegram, ни `AssignmentRule`/round-robin) — массовая загрузка 300 строк не должна генерировать 300 уведомлений и не должна сбивать курсор round-robin. **Назначение из колонки файла не поддерживается** (нет надёжного способа резолвить значение ячейки в `User.id` — имя? email? чужая компания? заблокирован?) — все импортированные лиды получают `assignedToId = null`, администратор распределяет вручную после импорта.

## Отчёт

### Поведение (FR-140-4)

```json
{
  "batchId": "clxxx",
  "totalRows": 315,
  "imported": 312,
  "skipped": 3,
  "duplicates": 18,
  "errors": 3
}
```

Отчёт виден сразу после импорта и позже — на странице `/admin/import` (история прошлых импортов).

## Откат импорта

### Поведение (FR-140-5)

`POST /api/import/:batchId/rollback` — удаляет все лиды этого батча (каскадно тянет их комментарии/события/задачи/напоминания/дубль-пометки), `ImportBatch.status = ROLLED_BACK`.

```
Если → batch.status !== "DONE"
То   → 400 (уже отменён или ещё не завершён)

Если → часть импортированных лидов уже изменена руками (комментарий, смена этапа) после импорта
То   → откат всё равно удаляет их — предупреждение в UI перед подтверждением отката:
       "Будет удалено N лидов, включая те, с которыми уже велась работа"
```

Откат — необратимое действие, требует явного подтверждения в UI (как удаление лида).

## Ограничения

| Параметр | Значение | Почему |
| --- | --- | --- |
| Максимум строк за один импорт | 5 000 | Файл обрабатывается в одном запросе без файлового хранилища — нужен предел на размер обработки за раз |
| Максимальный размер файла | 10 МБ | Разумный потолок для табличных данных такого объёма |
| Превышение | `400 FILE_TOO_LARGE` / `400 TOO_MANY_ROWS` с подсказкой разбить файл на части | — |

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
| --- | --- | --- | --- | --- |
| POST | `/api/import/preview` | Парсинг файла / превью с маппингом и дублями | Session | ADMIN only |
| POST | `/api/import/confirm` | Подтвердить и создать лиды | Session | ADMIN only |
| GET | `/api/import` | История импортов компании | Session | ADMIN only |
| POST | `/api/import/:batchId/rollback` | Откатить импорт | Session | ADMIN only |

### `POST /api/import/confirm`

**Request:** `{ "fileName": "leads.csv", "mapping": { "Имя клиента": "name", "Тел.": "phone", "Откуда": "customField:откуда" }, "rows": [ { ... } ] }`

`fileName` — обязательное поле (`ImportBatch.fileName` в БД не nullable); клиент передаёт `File.name`, сохранённое с шага загрузки. `confirmDuplicates` не входит в контракт — сервер никогда не ветвится по нему (дубли не блокируют создание нигде в продукте); это чисто клиентский чек-бокс подтверждения перед отправкой.

**Response 200:** `{ "batchId": "clxxx", "totalRows": 315, "imported": 312, "skipped": 3, "duplicates": 18, "errors": 3 }`

## Серверные правила безопасности

1. **Импорт — только ADMIN.** Менеджер не загружает массовые данные.
2. **Лимиты на размер файла и число строк — обязательны**, проверяются до начала парсинга, не после.
3. **`companyId` — из сессии.** Импортированные лиды принадлежат компании администратора, выполняющего импорт.
4. **Дедуп не блокирует** — как и везде, только помечает.
5. **Откат — каскадный и необратимый**, требует подтверждения.
6. **Мутации (`confirm`, `rollback`) — обычные проверки роли и видимости**, без отдельного guard'а подписки — биллинга в продукте нет.
7. **Импорт не запускает назначение и уведомления** — намеренное исключение из обычного флоу приёма, чтобы массовая загрузка не создавала шум и не сбивала round-robin.
