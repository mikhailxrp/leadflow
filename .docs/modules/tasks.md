# Модуль: Задачи (tasks)

> Спецификация создания, управления и выполнения задач по лидам. Задача привязана к лиду — при клике на задачу из любого места системы открывается карточка соответствующего лида.
> Связанные файлы: `.docs/database.md` (модели `Task`, `TaskStatus`, `EventType`), `.docs/modules/leads.md` (карточка лида, блок задач), `.docs/modules/admin-users.md` (управление менеджерами — исполнители задач).

## Цели модуля

После завершения этого модуля:

- Администратор и менеджер создают задачи по лидам с исполнителем и сроком
- Задачи видны в карточке лида
- Клик на задачу из любого места открывает карточку привязанного лида
- Менеджер меняет статус задачи (todo → in_progress → done)
- Администратор видит все задачи по всем лидам
- История изменений задачи пишется в `events`

**Не входит в модуль:**

- Глобальный список задач без привязки к лиду — out of scope MVP
- Уведомления о задачах (дедлайн приближается) — задел на будущее
- Шаблоны задач — out of scope MVP

## Чем задача отличается от напоминания

| | Напоминание | Задача |
|---|---|---|
| Кто создаёт | Сам себе | Кто угодно кому угодно |
| Цель | Уведомить в срок | Зафиксировать что нужно сделать |
| Исполнитель | Тот кто создал | Назначается явно |
| Статус | pending / fired | todo / in_progress / done |
| Контроль | Нет | Руководитель видит все задачи |
| Действие после срока | Автодоставка | Ничего — исполнитель сам меняет статус |

## Архитектурные решения

### 1. Задача всегда привязана к лиду

`Task.leadId` — обязательное поле, не nullable. Задачи вне контекста лида не существуют. Это намеренное ограничение MVP: задача = действие по конкретному клиенту.

### 2. Два пользователя: создатель и исполнитель

```prisma
createdById  String   // кто поставил задачу
assignedToId String   // кто должен выполнить
```

Это разные поля и разные роли. Руководитель ставит задачу менеджеру. Оба видят задачу в карточке лида.

### 3. completedAt фиксируется автоматически

При переводе статуса в `DONE` сервер сам выставляет `completedAt = now()`. Клиент не передаёт эту дату.

### 4. История через events

Каждое изменение задачи (создание, смена статуса, переназначение) → событие в таблицу `events` через `lib/events.ts`. История задачи читается из тех же событий что и история лида.

### 5. Каскад с лидом

`onDelete: Cascade` — при удалении лида удаляются все его задачи. Задача без лида не имеет смысла.

## Создание задачи

### Кто может создавать

- **ADMIN** — любому лиду, любому менеджеру
- **MANAGER** — только лидам в пределах своей видимости (`visibilityWhere`)

### Поля при создании

| Поле | Обязательное | Описание |
|---|---|---|
| `title` | Да | Что нужно сделать |
| `assignedToId` | Да | Кому назначена |
| `dueDate` | Нет | Срок выполнения |
| `description` | Нет | Подробное описание |

### Флоу

```
Кнопка «+ Добавить задачу» в карточке лида
  → модалка: title (обязательно), assignedTo (селект менеджеров), dueDate (датапикер), description
  → POST /api/leads/:id/tasks
  → сервер: Task.create + writeEvent(TASK_CREATED)
  → задача появляется в карточке лида
```

## Управление задачами

### Смена статуса

Менеджер меняет статус своей задачи. Админ меняет статус любой задачи.

```
TODO → IN_PROGRESS → DONE
                  ↘ CANCELLED (отменить)
```

```
PATCH /api/leads/:leadId/tasks/:taskId { status }
  → если status = DONE → completedAt = now()
  → если status = CANCELLED → completedAt = null
  → writeEvent(TASK_DONE | TASK_UPDATED | TASK_CANCELLED)
```

### Редактирование

`PATCH /api/leads/:leadId/tasks/:taskId` — изменить `title`, `description`, `dueDate`, `assignedToId`.

```
Если → status === DONE или CANCELLED
То   → редактирование запрещено → 400 TASK_NOT_EDITABLE
```

### Удаление

`DELETE /api/leads/:leadId/tasks/:taskId` — физическое удаление. Только ADMIN.

## Связь задача → лид

**Это ключевая навигационная связь модуля.**

Задача всегда знает свой `leadId`. При клике на задачу из любого места системы — переход на карточку лида.

### Где могут отображаться задачи (текущий MVP)

**Карточка лида** — основное место. Блок «Задачи» в правой колонке карточки.

### Навигация

```typescript
// Клик на задачу → переход на карточку лида
// Если уже на карточке лида — скролл к блоку задач
// Если в другом месте — router.push(`/leads/${task.leadId}`)

// компонент TaskItem.tsx
<div onClick={() => router.push(`/leads/${task.leadId}`)}>
  {task.title}
</div>
```

### Что показывается при переходе

Открывается карточка лида (`/leads/:id`). Если задача активна — блок задач в карточке подсвечивает конкретную задачу (через `?taskId=` в URL или через состояние).

## Отображение задач

### Блок «Задачи» в карточке лида

Расположен в правой колонке карточки, после блока «Напоминания».

```
┌─────────────────────────────────────────────┐
│ Задачи                           [+ Добавить] │
│                                              │
│ ○ Отправить КП                              │
│   Алексей · до 15 июня            [TODO]    │
│                                              │
│ ◉ Провести демо                             │
│   Мария · до 20 июня         [IN_PROGRESS]  │
│                                              │
│ ✓ Согласовать договор                       │
│   Алексей · выполнено 10 июня    [DONE]     │
└─────────────────────────────────────────────┘
```

**Сортировка:**
- Активные (`TODO`, `IN_PROGRESS`) — сверху, по `dueDate asc`
- Просроченные (dueDate < now, статус не DONE) — выделены красным
- Выполненные и отменённые — внизу, свёрнуто

### Бейджи статусов

| Статус | Цвет |
|---|---|
| TODO | Серый |
| IN_PROGRESS | Синий `#3B82F6` |
| DONE | Зелёный `#10B981` |
| CANCELLED | Серый, зачёркнут |

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
|---|---|---|---|---|
| GET | `/api/leads/:id/tasks` | Список задач по лиду | Session | Видимость по роли |
| POST | `/api/leads/:id/tasks` | Создать задачу | Session | ADMIN / MANAGER |
| PATCH | `/api/leads/:id/tasks/:taskId` | Изменить / сменить статус | Session | Автор или ADMIN |
| DELETE | `/api/leads/:id/tasks/:taskId` | Удалить задачу | Session | ADMIN only |

### `POST /api/leads/:id/tasks`

**Request:**
```json
{
  "title": "Отправить коммерческое предложение",
  "assignedToId": "clxxx",
  "dueDate": "2026-06-20T10:00:00Z",
  "description": "Включить тариф Про и примеры кейсов"
}
```

**Response 200:**
```json
{
  "id": "clyyy",
  "title": "Отправить коммерческое предложение",
  "status": "TODO",
  "assignedTo": { "id": "clxxx", "name": "Алексей" },
  "dueDate": "2026-06-20T10:00:00Z",
  "leadId": "clzzz",
  "createdAt": "2026-06-11T08:00:00Z"
}
```

### `PATCH /api/leads/:id/tasks/:taskId`

**Request (смена статуса):** `{ "status": "DONE" }`
**Request (редактирование):** `{ "title": "...", "dueDate": "...", "assignedToId": "..." }`

**Response 400:**
```json
{ "success": false, "error": "TASK_NOT_EDITABLE" }
```

## Серверные правила безопасности

1. **`companyId` из сессии.** Лид проверяется по `companyId` пользователя — нельзя создать задачу на лид чужой компании.

2. **Видимость задач через видимость лида.** `visibilityWhere` из `.docs/modules/leads.md` применяется к лиду прежде чем отдать его задачи — менеджер в режиме `OWN` не видит задачи чужого лида.

3. **Редактирование — автор или ADMIN.** Менеджер не редактирует чужие задачи.

4. **Удаление — только ADMIN.** Менеджер не удаляет задачи.

5. **`completedAt` — только сервер.** Клиент не передаёт дату выполнения — сервер выставляет сам при `status = DONE`.

6. **`assignedToId` — активный менеджер той же компании.** Нельзя назначить задачу заблокированному менеджеру или пользователю чужой компании.

7. **Редактирование DONE/CANCELLED запрещено.** Завершённая задача — финальное состояние.

8. **События пишет только сервер** через `lib/events.ts` при каждом изменении.
