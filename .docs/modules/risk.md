# Модуль: Индикатор риска (risk)

> Сквозная логика определения цвета риска и его причины для лида. Новый модуль — выделен отдельно, потому что используется четырьмя разными местами интерфейса и не должен реализовываться четыре раза по-разному.
> Связанные файлы: `.docs/database.md` (`Lead`, `Event`, `Task`, `PipelineStage`, `Company.settings`), `.docs/modules/leads.md`, `.docs/modules/pipeline.md`, `.docs/modules/today.md`, `.docs/modules/notifications.md` (использует те же нормативы).

## Цели модуля

После завершения этого модуля:

- Любой лид имеет вычисляемый цвет риска (зелёный/жёлтый/красный/серый) и текстовую причину
- Логика вычисления — в одном месте, используется списком, карточкой, доской, «Сегодня» и «Контролем»
- Риск не хранится в БД — вычисляется на момент запроса, как и дата смены статуса
- Расчёт учитывает: первый ответ, время на этапе, наличие следующего действия, наличие ответственного, статус закрытия

**Не входит в модуль:**

- Отображение бейджа (цвет, тултип) — в каждом потребителе своё, см. соответствующий модуль
- Эскалация и алерты на основе риска — `.docs/modules/notifications.md` (этот модуль только вычисляет, не уведомляет)
- Хранение нормативов — `.docs/modules/app-settings.md` (этот модуль только читает)

## Архитектурные решения

### 1. Чистая функция, без побочных эффектов

```typescript
// lib/risk/computeRisk.ts
export type RiskLevel = "green" | "yellow" | "red" | "grey";
export type RiskResult = { level: RiskLevel; reason: string | null };

export function computeRisk(input: RiskInput): RiskResult;
```

Никаких записей в БД, никаких уведомлений — только вычисление. Это позволяет вызывать функцию сколько угодно раз (в списке на 50 строк, в одиночной карточке, в cron-проверке) без побочных эффектов.

### 2. Один приоритет причин для всех потребителей

Если у лида одновременно верно несколько условий риска (например, нет ответственного И нет следующего действия), показывается **одна** причина — самая значимая. Без единого порядка приоритета разные экраны показывали бы разные причины для одного и того же лида, что подрывает доверие к индикатору.

### 3. Не пересчитывает то, что уже посчитано в `events`

Дата последнего `STAGE_CHANGED`, факт наличия `LEAD_TAKEN_IN_WORK` — читаются из `events`, не задаются заново. Risk-модуль — это слой интерпретации существующих данных, не источник новых.

## Цветовая схема и приоритет причин

| Приоритет | Условие | Цвет | Пример причины |
| --- | --- | --- | --- |
| 1 (высший) | `closeType` задан | **серый** | «Закрыт: сделка» / «Закрыт: отказ» |
| 2 | `assignedToId` отсутствует | **красный** | «Нет ответственного» |
| 3 | Нет `LEAD_TAKEN_IN_WORK`, прошло > 100% применимого норматива реакции | **красный** | «Нет первого ответа 38 минут» |
| 4 | Время на текущем этапе > лимита этапа/компании | **красный** | «7 дней на этапе» |
| 5 | Нет открытой задачи (следующего действия) | **жёлтый** | «Нет следующего шага» |
| 6 | Открытая задача просрочена (`dueDate < now`) | **жёлтый** | «Просрочена задача: {название}» |
| 7 | Нет `LEAD_TAKEN_IN_WORK`, прошло 66–100% норматива | **жёлтый** | «Приближается срок первого ответа» |
| — (нет совпадений выше) | — | **зелёный** | — (причина не показывается) |

**Серый и красный «нет ответственного» — высший приоритет** не случайно: это прямое отражение правила продукта «активного лида без ответственного не существует» (`.docs/modules/assignment.md`) — если это произошло, остальные сигналы риска вторичны.

## Алгоритм вычисления

```typescript
// lib/risk/computeRisk.ts — концепт
function computeRisk(input: RiskInput): RiskResult {
  if (input.lead.closeType) {
    return { level: "grey", reason: input.lead.closeType === "WON" ? "Закрыт: сделка" : "Закрыт: отказ" };
  }
  if (!input.lead.assignedToId) {
    return { level: "red", reason: "Нет ответственного" };
  }

  const norm = resolveApplicableNorm(input); // см. раздел 5
  const minutesSinceCreated = diffMinutes(now(), input.lead.createdAt, norm.workHoursOnly ? input.workHours : null);

  if (!input.hasTakenInWork) {
    if (minutesSinceCreated > norm.defaultMinutes) {
      return { level: "red", reason: `Нет первого ответа ${formatMinutes(minutesSinceCreated)}` };
    }
    if (minutesSinceCreated > norm.defaultMinutes * (norm.reminderBeforePercent / 100)) {
      return { level: "yellow", reason: "Приближается срок первого ответа" };
    }
  }

  const stageLimit = input.currentStage.stageTimeLimitDays ?? input.companySettings.stageStuckDaysDefault;
  const daysOnStage = diffDays(now(), input.lastStageChangedAt);
  if (daysOnStage > stageLimit) {
    return { level: "red", reason: `${daysOnStage} дней на этапе` };
  }

  if (input.overdueOpenTask) {
    return { level: "yellow", reason: `Просрочена задача: ${input.overdueOpenTask.title}` };
  }
  if (!input.hasOpenTask) {
    return { level: "yellow", reason: "Нет следующего шага" };
  }

  return { level: "green", reason: null };
}
```

## Применимый норматив реакции

Норматив переопределяется по трём осям (`Company.settings.reactionNorms`, см. `.docs/database.md`), порядок применения — **сотрудник важнее этапа важнее источника важнее общего дефолта**:

```
resolveApplicableNorm(lead):
  byUser[lead.assignedToId]      — если задан, выигрывает
  byStage[lead.stageId]          — если задан и нет переопределения по сотруднику
  bySource[sourceKey]            — если задан и нет переопределения выше
  defaultMinutes                  — иначе

sourceKey = lead.source, кроме 'manual'/'api' (ручное добавление и универсальный
вебхук) — они делят один общий ключ 'other' (constants/leadSources.ts,
OTHER_SOURCE_TYPES/OTHER_BYSOURCE_KEY), не переопределяются по отдельности.
```

`workHoursOnly` — общий флаг компании, применяется **только к нормативу первого ответа** (минутная точность чувствительна к ночным разрывам). Лимит «зависания» на этапе (дни) не корректируется по рабочим часам — разница в часах на горизонте нескольких дней не значима.

> **Статус реализации:** сейчас `lib/risk/workHoursUtils.ts` — осознанный no-op (флаг принимается, но не влияет на расчёт минут). Собственно вычитание нерабочих часов и настройка `Company.settings.workHours`/`reactionNorms` через `/admin/settings` реализуются в **Phase 17**; этот модуль лишь читает уже сохранённые значения, когда они появятся.

## Производительность на списках

Вызывать `computeRisk` построчно с отдельным запросом `events` на каждый лид в списке из 50+ строк — N+1-проблема. Вместо этого для списков/доски:

```
1. Один запрос: последние LEAD_TAKEN_IN_WORK / STAGE_CHANGED по всем нужным leadId (group by leadId)
2. Один запрос: открытые задачи (Task, status IN (TODO, IN_PROGRESS)) по тем же leadId, агрегат MIN(dueDate)
3. computeRisk() вызывается в памяти на собранных данных — без обращения к БД внутри цикла
```

Это деталь реализации `lib/risk/computeRiskBatch.ts`, а не другой алгоритм — результат идентичен поштучному вызову `computeRisk`.

## Серверные правила безопасности

1. **Риск вычисляется на сервере**, не на клиенте — клиент не должен получать сырые `events`, чтобы посчитать риск сам; ответ API уже содержит готовый `{ level, reason }`.
2. **Не хранится** — нет поля `Lead.risk`. Любой код, который пытается закэшировать риск в БД дольше длительности одного запроса, нарушает архитектурный принцип «риск вычисляется, а не хранится» (`prd.md` → принцип 12).
3. **Пакетный расчёт не должен N+1-запрашивать БД** — см. раздел 6.
