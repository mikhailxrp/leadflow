## Возможный пересмотр ролевой модели (отложено, НЕ отклонено)

**Когда:** вероятно вернёмся после первых тестов приложения.

**Текущее каноничное состояние (НЕ менять без отдельного решения):**
`ROLE_RANK = { MANAGER: 0, HEAD: 1, ADMIN: 2 }` — **ADMIN высший**. Управление пользователями и настройками — компетенция ADMIN. См. `CLAUDE.md`, `.docs/prd.md` §3, `.docs/database.md`.

**Предложенное будущее перераспределение ролей:**

- **Руководитель (HEAD) — самый высокий приоритет, выше Администратора.**
- **Только один Руководитель на компанию** (новый инвариант уникальности).
- Только Руководитель назначает/снимает Администраторов; пишет комментарии пользователям и ставит им задачи.
- Администратор управляет пользователями и настройками, но **не может назначить ADMIN** — это право только Руководителя.

**Почему отложили:** на этапе планирования Phase 10 решили пока оставить как есть (ADMIN высший), чтобы не делать кросс-каттинг рефакторинг до первых тестов. Идея отражает реальную бизнес-иерархию клиента (владелец/руководитель — главный, «администратор» — настройщик системы ниже него).

**Реализуемо. Точки правок, когда вернёмся:**

- `constants/roles.ts` — `ROLE_RANK` (поднять HEAD выше ADMIN) + при необходимости отдельная проверка «назначить ADMIN может только HEAD».
- `proxy.ts` — пороги доступа к `/admin/*`, `/control`, `/reports`.
- Все вызовы `hasMinRole(...)` — пересмотреть смысл порогов.
- Тексты: `.docs/prd.md` §3, `CLAUDE.md` (блок ролей), `.docs/database.md` (enum/ROLE_RANK, описания) — синхронизировать.
- Новый инвариант «один активный HEAD на компанию» — app-level проверка или partial unique index.
- API `/api/users` (Phase 10) — пересмотреть «кто кому назначает роль».

> Связано с Phase 10 (Админка: пользователи) — там реализуется управление ролями на текущей модели. План фазы: `.docs/phases/phase-10.md`.

**1 — Отдельного бота-программу писать не нужно**
«Бот» в Telegram — это не отдельный сервер, который надо где-то крутить. Это две вещи:

Идентичность бота — создаётся один раз вручную через @BotFather в Telegram (5 минут: имя + username → получаешь TELEGRAM_BOT_TOKEN). Никакого кода на этом шаге.
Мозги бота — это наше же приложение. Telegram пересылает входящие сообщения (нажатие /start) на наш эндпоинт /api/telegram/webhook, а мы отправляем уведомления обычным HTTP-запросом к Bot API (lib/telegram.ts). Отдельного процесса/деплоя нет.
Что реально «пишем» в Phase 13 — это webhook-обработчик (таск 2) и примитив отправки (таск 1). Плюс один ручной ops-шаг: вызвать setWebhook, чтобы Telegram знал наш URL (как crontab в Phase 1).

Итого: 1 общий бот на всю платформу, созданный в BotFather один раз; вся логика — внутри существующего Next.js-приложения.

2 — Настройка уведомлений: два уровня + привязка

Уровень компании (ADMIN) → тумблер telegramEnabled в /admin/settings
↓ (если включён)
Привязка пользователя → /profile → «Подключить Telegram»
→ ссылка t.me/<bot>?start=<token> → жмёт Start
→ сохраняется его telegramChatId
↓ (если привязан)
Личные настройки (сам юзер) → тумблеры в /profile: «Новый лид назначен на меня» и т.д.
Уведомление уходит только когда совпали все три: компания включила Telegram И у пользователя привязан chatId И его личный тумблер типа не выключен.

Важные детали:

Каждый пользователь настраивает себя сам на /profile (self-service) — администратор не вводит чужой chatId руками (это и есть смысл замены старого мока с ручным «Chat ID»).
Настройки персональные: один менеджер может отключить Telegram-уведомления, другой — оставить, не влияя друг на друга.
В Phase 13 живой тип один — «новый лид назначен на меня» (→ назначенному менеджеру). Остальные тумблеры (комментарии, напоминания) получат реальные триггеры в следующих фазах (14, 17).
Хочешь — добавлю этот флоу как короткую диаграмму в сам phase-13.md, чтобы при реализации не восстанавливать по памяти.

**Переменные в .env**

TELEGRAM_BOT_TOKEN= # токен бота от @BotFather — уже есть в .env, но сейчас пустой
TELEGRAM_BOT_USERNAME= # username бота без @ (например leadkanal_bot) — для deep-link в /api/telegram/bind

TELEGRAM_WEBHOOK_SECRET= # произвольная случайная строка — секрет для проверки заголовка X-Telegram-Bot-Api-Secret-Token в /api/telegram/webhook

TELEGRAM_BOT_TOKEN — уже объявлена в .env, просто сейчас пустая (без неё sendTelegramMessage тихо no-op'ится, лид не теряется, но сообщения не уходят).

TELEGRAM_BOT_USERNAME и TELEGRAM_WEBHOOK_SECRET — ещё не добавлены; они появятся в CLAUDE.md официально в Task 3, но фактически нужны в .env уже сейчас, так как код (Task 2) их читает.
Ручной шаг (не код, не .env)
Нужно один раз зарегистрировать вебхук у самого Telegram, указав секрет:

curl -X POST "https://api.telegram.org/bot<TELEGRAM_BOT_TOKEN>/setWebhook" \
 -d "url=https://<ваш-домен>/api/telegram/webhook" \
 -d "secret_token=<TELEGRAM_WEBHOOK_SECRET>"
Без setWebhook Telegram не будет слать апдейты на /api/telegram/webhook, и /start <token> в боте не привяжет chat_id, даже если все переменные в .env заполнены.

**Настройка напоминаний CronTab**

Можно настроить прямо здесь — необязательно лезть на VPS по SSH. Команда — это просто HTTP-запрос на публичный URL приложения, поэтому неважно, на каком физическом сервере крутится cron: лишь бы у него был доступ в интернет.

_Поля (сейчас там "1 1 _ \* " — это конкретное время 01:01 раз в сутки, надо на «каждую минуту»):

Поле Значение
Минуты _
Часы _
Дни _
Месяцы _
Дни недели \*
Команда (важно — cron не читает .env приложения, поэтому $CRON_SECRET и домен нужно вписать реальными значениями, а не как переменные):

`curl -s -X POST -H "Authorization: Bearer РЕАЛЬНЫЙ_CRON_SECRET" "https://ваш-домен/api/cron/reminders" >/dev/null`

_Где:_
РЕАЛЬНЫЙ_CRON_SECRET — значение из .env вашего сервера (CRON_SECRET=), скопировать как есть
ваш-домен — APP_URL прод-окружения
Краткое описание: что угодно понятное, например «Напоминания по лидам».

Один нюанс: этот эндпоинт (/api/cron/reminders) ещё не реализован — он появится в Таске 1 фазы 14. Настраивать это задание в планировщике имеет смысл после того, как код будет задеплоен, иначе curl будет бить в несуществующий роут (получит 404, ничего страшного не сломает, но и толку пока нет).
