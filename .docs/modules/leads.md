## Цели модуля

После завершения этого модуля:

- Пользователь видит список лидов (все или только свои — по настройке)
- Работает поиск по имени, телефону, email
- Работают фильтры: источник, менеджер, статус, период
- Открывается карточка лида со всеми данными
- Видны маркетинговые данные, UTM, кастомные поля формы
- Менеджер добавляет комментарии
- Видна полная история изменений лида
- Контактные данные лида можно редактировать
- Админ может удалить лид

**Не входит в модуль:**

- Приём лидов → `.docs/modules/leads-intake.md`
- Kanban-воронка и drag-and-drop → `.docs/modules/pipeline.md`
- Логика автораспределения → `.docs/modules/assignment.md`
- Уведомления о новых лидах → `.docs/modules/notifications.md`

## Видимость лидов (право доступа)

Видимость менеджерами настраивается админом — глобальный переключатель `Company.settings.leadVisibility` (`ALL | OWN`).

```
Если → role === ADMIN
То   → видит все лиды компании (всегда)

Если → role === MANAGER и leadVisibility === ALL
То   → видит все лиды компании

Если → role === MANAGER и leadVisibility === OWN
То   → видит только лиды, где assignedToId === его id
```

**Это правило применяется на сервере в каждом запросе списка и карточки.** Менеджер не может открыть чужой лид по прямому URL, если режим `OWN` — сервер вернёт 404.

```typescript
// lib/leads/visibilityFilter.ts — концепт
function visibilityWhere(session): Prisma.LeadWhereInput {
  const base = { companyId: session.user.companyId };       // всегда изоляция тенанта
  if (session.user.role === "ADMIN") return base;
  const settings = ...; // settings компании
  if (settings.leadVisibility === "OWN")
    return { ...base, assignedToId: session.user.id };
  return base;
}
```

## Список лидов

### Отображение

Таблица (`components/leads/LeadsTable.tsx`). Колонки: Имя, Телефон, Email, Источник, Менеджер, Статус, Дата.

- Строка кликабельна → открывает карточку
- Источник — бейдж с иконкой и цветом (см. `design-system.md` → бейджи источников)
- Статус — цветной бейдж этапа воронки
- Пустое состояние (нет лидов / нет результатов фильтра) — `components/leads/EmptyState.tsx`

### Пагинация

Серверная, через `skip`/`take` в Prisma. Размер страницы — 20 (константа). Параметры `page`, `pageSize` в query. Ответ включает `total` для отрисовки пагинации.

### Сортировка

По умолчанию — `createdAt desc` (новые сверху). В MVP сортировка фиксирована; настраиваемая — задел на будущее.

## Поиск и фильтрация

### Поиск (FR-40)

По `name`, `phone`, `email` — частичное совпадение, регистронезависимо.

```typescript
// частичное совпадение по трём полям одновременно
where.OR = [
  { name:  { contains: q, mode: "insensitive" } },
  { phone: { contains: q } },
  { email: { contains: q, mode: "insensitive" } },
];
```

Все три поля проиндексированы (`.docs/database.md` → «Индексы»).

### Фильтры (FR-41…FR-44)

| Фильтр | Поле | Параметр query |
| --- | --- | --- |
| Источник лида | `source` | `?source=TILDA` |
| Ответственный менеджер | `assignedToId` | `?manager=<userId>` |
| Статус воронки | `stageId` | `?stage=<stageId>` |
| Период поступления | `createdAt` (диапазон) | `?from=<date>&to=<date>` |

### Комбинирование (FR-45)

Поиск и фильтры работают одновременно — собираются в единый `where`, поверх него накладывается `visibilityWhere`. Порядок сборки:

```
where = visibilityWhere(session)        // изоляция тенанта + видимость менеджера
      + поиск (OR по трём полям)
      + активные фильтры (source, manager, stage, период)
```

Фильтрация — на сервере (`GET /api/leads`), не на клиенте. Клиент только собирает query-параметры.

## Карточка лида

Открывается по клику на строку списка или карточку в Kanban. Может быть модалкой или отдельной страницей `/leads/:id`.

### Структура (два столбца)

**Левый столбец — данные лида:**

- **Контакты:** имя, телефон, email, комментарий, дата создания
- **Маркетинг:** источник (бейдж), UTM-метки (таблица key→value, моноширинный шрифт), реферер, landing page
- **Данные Яндекс Директа** (если есть в `marketing`): кампания, группа, ключевая фраза, устройство, регион
- **Дополнительные поля** (`customFields`): все нестандартные поля формы как «ключ: значение»

**Правый столбец — работа с лидом:**

- **Статус** — текущий этап (селект или степпер); смена → `.docs/modules/pipeline.md`
- **Ответственный** — селект менеджера; смена → `.docs/modules/assignment.md`
- **Комментарии** — лента + поле добавления
- **История изменений** — таймлайн событий

### Источник данных карточки

`GET /api/leads/:id` возвращает лид + комментарии + события. Применяется `visibilityWhere`: менеджер в режиме `OWN` получит 404 на чужой лид.

## Комментарии

### Отображение (FR-14)

Лента в карточке: аватар-инициалы автора + имя + дата + текст. Сортировка — старые сверху (хронология обсуждения).

### Добавление

`POST /api/leads/:id/comments` с текстом. Сервер:

1. Проверяет доступ к лиду (`visibilityWhere`)
2. Создаёт `Comment` (companyId косвенно через лид, `userId` = автор из сессии)
3. Пишет событие `COMMENTED` через `lib/events.ts`
4. Возвращает созданный комментарий

Менеджер и админ могут комментировать. Редактирование/удаление комментариев — out of scope MVP.

---

## История изменений

### Отображение (FR-15, FR-26)

Таймлайн в карточке. Источник — таблица `events`, выборка по `leadId`, сортировка `createdAt desc`.

Каждое событие рендерится в человекочитаемую строку по `type` + `payload`:

| `type` | Строка в таймлайне |
| --- | --- |
| `LEAD_CREATED` | «Лид создан из {source}» |
| `STAGE_CHANGED` | «Статус: {fromStage} → {toStage}» |
| `ASSIGNED` | «Назначен менеджер: {name}» |
| `COMMENTED` | «Добавлен комментарий» |
| `LEAD_UPDATED` | «Изменены данные: {поля}» |

Имена этапов и менеджеров резолвятся по id из payload. Рендер-маппинг — в `constants/eventLabels.ts`.

### Источник правды

История — **только из `events`**, не из отдельных флагов. Это Event Sourcing: добавили новый тип события → он появляется в истории без изменения схемы лида.

## Редактирование лида

Контактные данные (имя, телефон, email, комментарий) редактируются в карточке (FR — карточка). `PATCH /api/leads/:id`.

Сервер:

1. Проверяет доступ (`visibilityWhere`)
2. Обновляет разрешённые поля
3. Пишет событие `LEAD_UPDATED` с payload `{ changed: [поля] }`

**Что нельзя менять через этот эндпоинт:** `source`, `customFields`, `utm`, `marketing` (это данные приёма, неизменны), `companyId`. Смена статуса и менеджера — отдельные действия (свои эндпоинты/модули).

Менеджер и админ могут редактировать контакты. (Если в режиме `OWN` — только свои лиды.)

---

## Удаление лида

Только админ (FR — роли). `DELETE /api/leads/:id`.

- UI — модалка подтверждения «Удалить лид? Действие необратимо»
- Каскадно удаляются `Comment` и `Event` лида (см. `.docs/database.md` → связи)
- Перед удалением пишется событие `LEAD_DELETED`? — **нет**, событие удалится каскадом. Запись об удалении идёт в общий журнал действий компании (событие без `leadId`, payload содержит снимок ключевых полей удалённого лида)

```
Если → role !== ADMIN
То   → 403, удаление запрещено
```

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
| --- | --- | --- | --- | --- |
| GET | `/api/leads` | Список с поиском, фильтрами, пагинацией | Session | Видимость по роли |
| GET | `/api/leads/:id` | Карточка (лид + комментарии + события) | Session | Видимость по роли |
| POST | `/api/leads` | Создать лид вручную | Session | ADMIN / MANAGER |
| PATCH | `/api/leads/:id` | Изменить контактные данные | Session | Видимость по роли |
| DELETE | `/api/leads/:id` | Удалить лид | Session | ADMIN only |
| POST | `/api/leads/:id/comments` | Добавить комментарий | Session | Видимость по роли |
| GET | `/api/leads/:id/events` | История изменений | Session | Видимость по роли |

### `GET /api/leads`

**Request:**

```
GET /api/leads?q=иван&source=TILDA&stage=clxxx&from=2026-06-01&to=2026-06-07&page=1&pageSize=20
```

**Response 200:**

```json
{
  "leads": [
    { "id": "clxxx", "name": "Иван Петров", "phone": "+7...", "email": "ivan@mail.ru",
      "source": "TILDA", "stageId": "clyyy", "assignedTo": { "id": "...", "name": "Алексей" },
      "createdAt": "2026-06-07T10:00:00Z" }
  ],
  "total": 248,
  "page": 1,
  "pageSize": 20
}
```

### `POST /api/leads/:id/comments`

**Request:** `{ "text": "Перезвонил, ждёт КП" }`

**Response 200:**

```json
{ "id": "clzzz", "text": "Перезвонил, ждёт КП", "user": { "id": "...", "name": "Алексей" }, "createdAt": "..." }
```

### `DELETE /api/leads/:id`

**Response 200:** `{ "success": true }`
**Response 403 (не админ):** `{ "success": false, "error": "FORBIDDEN" }`

## Серверные правила безопасности

1. **`visibilityWhere` применяется в КАЖДОМ запросе лидов** — список, карточка, комментарии, история, редактирование. Никогда не запрашивай лид без него.

2. **`companyId` всегда из сессии.** Менеджер не может передать чужой `companyId` в query, чтобы увидеть лиды другого тенанта.

3. **Менеджер в режиме `OWN` получает 404 на чужой лид**, не 403 — не раскрываем сам факт существования лида.

4. **Экранирование `customFields` и `comment` при выводе.** Это сырые данные формы (могут содержать XSS). React экранирует по умолчанию, но при `dangerouslySetInnerHTML` — запрещено. Выводим только как текст.

5. **Удаление — только ADMIN.** Проверка роли на сервере, не на клиенте.

6. **`PATCH` не меняет данные приёма** (`source`, `utm`, `marketing`, `customFields`) и `companyId`. Только контактные поля.

7. **События пишет сервер** через `lib/events.ts` при каждом изменении (комментарий, правка, удаление).

8. **Пагинация обязательна** — `GET /api/leads` никогда не возвращает все лиды без `take`. Защита от выгрузки всей базы одним запросом.
