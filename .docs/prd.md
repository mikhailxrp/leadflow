## 1. Обзор продукта

Мини-CRM — веб-приложение для малого бизнеса для централизованного приёма, хранения и обработки лидов из рекламных каналов. Документ детализирует требования к MVP: функциональность, схему данных, API, экраны и нефункциональные требования.

### 1.1 Проблема и решение

Малый бизнес теряет заявки: они приходят из разных каналов (Tilda, WordPress, Яндекс Директ, лендинги), нигде не собираются вместе, менеджеры не видят полной картины и не контролируют статус работы с клиентом.

CRM решает это: единая точка приёма всех лидов, автоматическое сохранение маркетинговых данных, воронка продаж, разграничение прав и уведомления в реальном времени.

### 1.2 Цели MVP

- Принимать лиды из любых источников через webhook и API
- Хранить все поля формы без потерь, включая новые поля без доработки
- Вести лид по настраиваемой воронке продаж
- Распределять лиды между менеджерами (авто + вручную)
- Уведомлять о новых лидах в интерфейсе и в Telegram
- Заложить архитектуру под аналитику и мультитенантность

### 1.3 Метрики успеха

|Метрика|Целевое значение|
|---|---|
|Потеря лидов|0% — каждая заявка сохранена|
|Время приёма лида|< 2 сек от webhook до отображения|
|Доставка уведомления|< 5 сек (интерфейс и Telegram)|
|Адаптация к новым полям формы|Без участия разработчика|

## 2. Модель поставки и лицензирование

### 2.1 Модель продукта

MVP поставляется по модели **Boxed (в одни руки)** — продукт для одного конкретного клиента, разовая поставка.

Архитектура при этом строится **мультитенантной** (поле `company_id` во всех таблицах). Это позволяет в будущем перейти на SaaS-модель (подписка, самостоятельная регистрация компаний) без переработки ядра — добавляется только слой регистрации и биллинга поверх готовой базы.

|Аспект|Решение для MVP|
|---|---|
|Модель продажи|Boxed, разовая поставка одному клиенту|
|Регистрация компаний|Отсутствует (создаётся при развёртывании)|
|Биллинг|Отсутствует|
|Архитектура|Мультитенантная (задел на SaaS)|
|Количество компаний|Одна (одна запись в `companies`)|

### 2.2 Поставка и развёртывание

- Продукт разворачивается на сервере клиента
- Развёртывание выполняет **поставщик** (не клиент)
- Поставляется **production-build** (собранное приложение), не исходный код
- Все секреты хранятся в `.env` на сервере, вне репозитория: `DATABASE_URL`, `AUTH_SECRET`, `LICENSE_KEY`, токен Telegram-бота

### 2.3 Защита от копирования

Цель защиты — убрать соблазн лёгкого копирования, а не построить абсолютную крепость. Основной барьер — юридический (договор). Технические меры дополняют его.

| ID     | Требование                                                                                                            |
| ------ | --------------------------------------------------------------------------------------------------------------------- |
| LIC-01 | Поле `licenseKey` в таблице `companies`                                                                               |
| LIC-02 | При старте приложение читает `LICENSE_KEY` из env и сверяет с хэшем; при несовпадении — не стартует (офлайн-проверка) |
| LIC-03 | Секреты вынесены в `.env`, не входят в поставку кода                                                                  |
| LIC-04 | Поставка в виде production-build, без исходников                                                                      |
| LIC-05 | Юридическая защита — лицензионный договор с запретом перепродажи и копирования                                        |

> **Важно:** технические меры не остановят профессионала с полным доступом к серверу. Они рассчитаны на то, чтобы исключить «копирование как есть». Реальную защиту даёт договор.

### 2.4 Первичная инициализация

Так как самостоятельной регистрации нет, первый администратор и компания создаются при развёртывании.

|ID|Требование|
|---|---|
|INIT-01|Сид-скрипт создаёт компанию (одна запись) при первом запуске|
|INIT-02|Сид-скрипт создаёт первого администратора из переменных окружения (email, пароль)|
|INIT-03|Сид-скрипт создаёт 5 дефолтных этапов воронки для компании|
|INIT-04|Повторный запуск сид-скрипта не создаёт дубликатов (идемпотентность)|

## 3. Пользователи и роли

В системе две роли. Видимость лидов менеджерами настраивается администратором (глобальный переключатель).

### 3.1 Администратор

Полный доступ. Управляет пользователями, интеграциями, воронкой и настройками системы.

|Возможность|Доступ|
|---|---|
|Просмотр всех лидов|Да|
|Создание и редактирование лидов|Да|
|Создание / блокировка / разблокировка / удаление менеджеров|Да|
|Настройка интеграций (Tilda, WordPress, Яндекс, webhook)|Да|
|Настройка этапов воронки|Да|
|Назначение ответственных менеджеров|Да|
|Настройка правил автораспределения|Да|
|Переключатель видимости лидов для менеджеров|Да|
|Просмотр истории изменений и журнала действий|Да|
|Удаление лидов|Да (с подтверждением)|

> Блокировка предпочтительнее удаления — сохраняется вся история действий менеджера. Удаление необратимо и требует подтверждения.

### 3.2 Менеджер

Операционная роль. Работает с лидами в рамках прав, заданных администратором.

|Возможность|Доступ|
|---|---|
|Просмотр лидов|Все или только свои (настройка админа)|
|Изменение статуса лида (drag-and-drop в воронке)|Да|
|Добавление комментариев|Да|
|Редактирование контактных данных лида|Да|
|Управление пользователями|Нет|
|Настройка интеграций|Нет|
|Настройка воронки|Нет|
|Удаление лидов|Нет|

## 4. Функциональные требования

Требования имеют ID для трассировки в фазах и тасках SDD (формат FR-XX).

### 4.1 Приём лидов

|ID|Требование|
|---|---|
|FR-01|Webhook-эндпоинт для приёма заявок Tilda со всеми полями формы|
|FR-02|Универсальный webhook `/api/webhooks/leads` — принимает любой JSON|
|FR-03|Аутентификация источника через API-ключ (заголовок или параметр `key`)|
|FR-04|Интеграция WordPress через webhook плагинов (CF7, WPForms, Gravity Forms)|
|FR-05|Сохранение нестандартных полей формы в JSONB без потерь|
|FR-06|Каждая заявка создаёт отдельный лид (дедупликация не выполняется)|
|FR-07|Фиксация источника по параметру `source` или привязке API-ключа|
|FR-08|Приём лида не блокируется при невалидных доп. полях — пишется как есть|

**Яндекс Директ (два режима).** Реализуются оба. Полный — приоритетный; минимальный — фолбэк, если доступ к API не получен.

|Режим|Что получаем|Когда|
|---|---|---|
|Полный (API)|Campaign ID, кампания, Ad Group, ключевая фраза, устройство, регион|Приоритет, если есть доступ к API|
|Минимальный (UTM)|Только UTM-метки из URL перехода|Фолбэк без доступа к API|

**Маркетинговые данные (все источники):**

- UTM-метки: `utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content`
- Реферер, landing page, источник трафика, дата и время создания

|ID|Требование|

|FR-12|Дополнительные поля формы (из JSONB) в виде «ключ: значение»|
|FR-13|Рабочие данные: ответственный менеджер, текущий статус воронки|

### 4.3 Воронка продаж (Kanban)

|ID|Требование|

|ID|Требование|
|---|---|
|FR-60|Отображение нового лида в интерфейсе в реальном времени (SSE)|
|FR-61|Toast-уведомление о новом лиде с кнопкой «Открыть»|
|FR-62|Telegram-уведомление ответственному менеджеру (в MVP)|
|FR-63|Привязка Telegram-аккаунта менеджера в профиле (`chat_id`)|
|FR-64|Счётчик непрочитанных уведомлений (колокольчик в хедере)|

### 4.8 Напоминания

|ID|Требование|
|---|---|
|FR-70|Менеджер или администратор устанавливает напоминание на конкретный лид с датой, временем и текстом|
|FR-71|В указанное время напоминание отправляется пользователю по выбранным каналам|
|FR-72|Каналы доставки в MVP: Telegram и Email (выбираются пользователем при создании)|
|FR-73|Напоминание привязано к лиду: в карточке лида видна лента активных и исполненных напоминаний|
|FR-74|Пользователь может редактировать и отменять напоминание до его срабатывания|
|FR-75|После срабатывания напоминание отмечается как выполненное (статус `fired`), не удаляется|
|FR-76|Архитектура поддерживает добавление новых каналов доставки (SMS, push и др.) без переработки ядра|

**Поведение при срабатывании:**

```
В указанное время:
  → Telegram: сообщение боту пользователя (если chat_id привязан)
  → Email: письмо на email пользователя
  → Оба канала могут быть выбраны одновременно
  → Сбой одного канала не отменяет доставку по другому
  → Напоминание переходит в статус fired независимо от успеха доставки
```

**Расширяемость каналов (FR-76):** список каналов хранится в `reminder.channels` (JSONB-массив: `["telegram", "email"]`). Добавление нового канала — новый обработчик в `lib/reminders/channels/`, без изменения схемы.

### 4.9 Задачи

|ID|Требование|
|---|---|
|FR-80|Менеджер или администратор создаёт задачу по лиду: название, исполнитель, срок, описание|
|FR-81|Задача всегда привязана к лиду — существует только в контексте лида|
|FR-82|Клик на задачу из любого места системы открывает карточку привязанного лида|
|FR-83|Задача имеет статус: TODO → IN_PROGRESS → DONE / CANCELLED|
|FR-84|При переводе в DONE сервер автоматически фиксирует дату выполнения (`completedAt`)|
|FR-85|В карточке лида отображается блок задач: активные сверху, выполненные и отменённые — свёрнуто внизу|
|FR-86|Просроченные задачи (dueDate прошёл, статус не DONE) выделяются визуально|
|FR-87|Администратор видит все задачи, менеджер — только задачи в пределах своей видимости лидов|

## 5. Модель данных

PostgreSQL + Prisma. Каждая таблица содержит `company_id` с первого дня для мультитенантности. Сейчас одна компания, расширение не требует миграций.

### 5.1 Схема таблиц

**companies — компании (тенанты)**

| Поле       | Тип         | Описание                                        |
| ---------- | ----------- | ----------------------------------------------- |
| id         | uuid PK     | Идентификатор компании                          |
| name       | text        | Название компании                               |

## 11. Ключевые архитектурные принципы

Сквозные принципы, на которых стоит вся система. Детали — в `CLAUDE.md` и модулях.

1. **Мультитенантность с первого дня.** Каждая таблица содержит `companyId`. Любой запрос к данным фильтруется `where: { companyId }` из сессии. Сейчас компания одна (Boxed), но код пишется так, будто их много — переход на SaaS не потребует переработки ядра. Запрос без фильтра по `companyId` считается багом (утечка между тенантами).
    
2. **Лид нельзя потерять.** Приём заявки не падает из-за структуры данных. Известные поля → колонки, UTM → `utm`, маркетинг → `marketing`, всё прочее → `customFields` (JSONB). Отклонить приём может только невалидный API-ключ или rate limit, но не неожиданный формат формы. См. `.docs/modules/leads-intake.md`.
    
3. **Настраиваемая воронка вместо хардкода.** Этапы — записи в `PipelineStage` (`name`, `color`, `order`), не enum в коде. Админ управляет ими через UI. У каждой компании своя воронка. Добавление/удаление этапа не требует миграции. См. `.docs/modules/pipeline.md`.
    
4. **Event Sourcing для истории и аналитики.** Все значимые действия (создание лида, смена этапа, назначение, комментарий) пишутся в единую таблицу `events`. История лида и будущая аналитика строятся поверх неё без изменения схемы. События пишет только сервер через `lib/events.ts` — у клиента нет API на запись. См. `.docs/database.md`.
    
5. **Серверная защита всех критичных операций.** Видимость лидов, роли, изоляция тенанта проверяются на сервере. Клиентские проверки можно обойти. `companyId` берётся только из сессии, никогда из тела запроса. Менеджер в режиме `OWN` получает 404 на чужой лид (не 403 — не раскрываем существование).
    
6. **Распределение лидов вне кода.** Режим (`MANUAL` / `ROUND_ROBIN`) — флаг в `Company.settings`, меняется админом без релиза. Round-robin защищён advisory lock от гонок при одновременном приёме. См. `.docs/modules/assignment.md`.
    
7. **Хранение настроек отделено от их применения.** `Company.settings` (JSONB) хранит флаги; применяют их соответствующие модули (`assignment` читает `assignMode`, `leads` — `leadVisibility`, `notifications` — `telegramEnabled`). Меняешь флаг → меняется поведение, код логики не трогается. См. `.docs/modules/app-settings.md`.
    
8. **Уведомления не блокируют приём.** SSE и Telegram отправляются после коммита транзакции приёма. Сбой Telegram (недоступен API) не откатывает сохранённый лид. Приём важнее доставки уведомления. См. `.docs/modules/notifications.md`.
    
9. **Ключи и пароли — только в виде хэшей.** API-ключи (`ApiKey.keyHash`) и пароли (`User.passwordHash`) хранятся хэшированными. Plain-значение API-ключа показывается один раз при создании. Секреты (`LICENSE_KEY`, `TELEGRAM_BOT_TOKEN`, OAuth Яндекса) — только в `.env` на сервере, вне репозитория.

## 14. Роадмап

Декомпозиция на фазы по принципу **vertical slices**: для каждой фичи, зависящей от контента/настроек, сначала админский CRUD, затем игровая/рабочая логика. Каждая фаза заканчивается рабочим, проверяемым результатом.

Фазы намеренно мелкие (1–3 дня). Внутри фазы — таски (1–2 дня, 3–7 файлов каждый) — расписываются агентом при инициации фазы через `.docs/prompts/phase-init.md`.

> **⚠️ О формулировке «Модель `X` в БД» ниже:** все Prisma-модели и enum'ы создаются в **Фазе 0** одной init-миграцией. В последующих фазах модели **не создаются заново** — пункты вида «API + UI для X» означают работу над уже существующей моделью. Новая миграция нужна только при изменении полей. Источник правды по схеме — `.docs/database.md`.

> **О деплое:** деплой и CI/CD вынесены в раннюю **Фазу 1** — чтобы каждая последующая фаза докатывалась на уже работающий прод. Развёртывание выполняется **вручную** (поставщик настраивает сервер клиента сам); CI/CD на GitHub Actions автоматизирует сборку и доставку при пуше в `main`.

### Фаза 0 — Фундамент

**Цель:** Каркас Next.js 16 + Prisma + PostgreSQL + базовая защита роутов. Никакого UI пока.

Что должно работать в конце:

- `npm run dev` запускает приложение, БД подключена
- Вся схема Prisma создана одной init-миграцией (все модели и enum'ы из `.docs/database.md`)
- `lib/license.ts` проверяет `LICENSE_KEY` при старте
- Сидер: компания + первый админ из ENV + 5 дефолтных этапов воронки (идемпотентно)
- `proxy.ts` редиректит на `/login` для защищённых роутов
- Базовый layout-каркас с заглушкой
- Health-check эндпоинт отвечает

### Фаза 1 — Деплой и CI/CD

**Цель:** Приложение работает на сервере клиента по реальному домену с SSL ещё с пустым каркасом. Каждая последующая фаза докатывается на уже работающий прод.

Что должно работать:

- Сервер клиента настроен вручную: Node.js, Nginx (reverse proxy), PM2 (процесс с автоперезапуском)
- SSL-сертификаты (Let's Encrypt через certbot)
- PostgreSQL на сервере подключена, миграции применяются (`prisma migrate deploy`)
- `.env` на сервере со всеми секретами (`DATABASE_URL`, `AUTH_SECRET`, `LICENSE_KEY`, `TELEGRAM_BOT_TOKEN`)
- GitHub Actions: pipeline сборки и деплоя при пуше в `main` (build → деплой на сервер → `migrate deploy` → перезапуск PM2)
- Nginx rate limit для эндпоинта логина: 10 запросов/мин на IP. Защита от brute force (NextAuth handler нельзя ограничить в коде). См. `.docs/modules/auth.md` → «Rate limiting».
- Smoke-test на пустом каркасе: health-check отвечает, `/dashboard` редиректит на `/login`

**Что НЕ делается в этой фазе:**

- Полные smoke-tests по фичам — нечего тестировать пока.

> Развёртывание сервера выполняется руками поставщиком. CI/CD автоматизирует только повторную доставку кода, не первичную настройку сервера.

### Фаза 2 — Авторизация (полный цикл)

**Цель:** Пользователь может войти по email + паролю, выйти. Заблокированный менеджер не входит. Роль пишется в сессию.

Что должно работать:

- Страница `/login`, форма входа
- NextAuth v5 credentials-провайдер, JWT-сессия 24 часа
- `role` и `companyId` в сессии (callbacks)

### Фаза 12.5 — Напоминания

**Цель:** Пользователь ставит напоминание по лиду на конкретную дату и время — в срок оно уходит в Telegram и/или Email.

Что должно работать:

- Миграция: таблица `reminders` (lead_id, created_by, text, remind_at, channels JSONB, status, fired_at)
- `lib/reminders/scheduler.ts` — планировщик: cron-джоб (каждую минуту) выбирает `pending` напоминания с `remind_at <= now()`, запускает доставку, выставляет `status = fired`
- `lib/reminders/channels/telegram.ts` — доставка в Telegram (через `lib/telegram.ts`)
- `lib/reminders/channels/email.ts` — доставка на Email (через nodemailer или SMTP-сервис из `.env`)
- Доставка по каналам параллельная (`Promise.allSettled`): сбой одного не отменяет другой
- При ошибке доставки — событие `REMINDER_FAILED` в `events` (аудит)
- CRUD API: `GET/POST /api/leads/:id/reminders`, `PATCH/DELETE /api/leads/:id/reminders/:rid`
- Блок «Напоминания» в карточке лида:
  - Список активных (`pending`) с датой и каналами
  - Кнопка «Добавить напоминание» — модалка: текст, дата+время (датапикер), чекбоксы Telegram / Email
  - Кнопка отмены (→ `status = cancelled`) до срабатывания
  - Исполненные (`fired`) показываются в истории напоминаний (свёрнуто)
- Валидация: `remind_at` не в прошлом, хотя бы один канал выбран
- `channels` — JSONB-массив (задел FR-76): новые каналы добавляются как handler в `lib/reminders/channels/` без изменения схемы

**Архитектурное решение по планировщику:** cron через `node-cron` в том же процессе Next.js (PM2). Для MVP достаточно, точность ±1 мин. Если нагрузка вырастет — вынести в отдельный worker или Bull/BullMQ (задел).

**ENV-переменные:** `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM` — для Email-канала.

См. `.docs/modules/reminders.md`.
