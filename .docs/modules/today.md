## Цели модуля

После завершения этого модуля:

- При входе пользователь видит `/today` — собранную картину того, что нужно сделать прямо сейчас, без поиска по спискам и фильтрам
- Экран персональный — у каждого пользователя своя «Сегодня», не общая компанийская сводка
- Все блоки — производные от уже существующих данных, без новых сущностей в БД

**Не входит в модуль:**

- Алгоритм риска → `.docs/modules/risk.md`
- Компанийская сводка зависших лидов для Руководителя/Администратора — это `/control`, не этот экран
- Создание/редактирование задач и лидов — переход в карточку, сама форма — в `tasks.md`/`leads.md`

---

## Архитектурные решения

### 1. Персональный экран, не управленческий

`/today` отвечает на вопрос «что нужно сделать **мне**». Компанийская картина рисков («сколько лидов под угрозой во всей компании») — это `/control`, отдельный экран для роли Руководитель/Администратор с другим вопросом. Если администратор сам ведёт лиды (назначены на него) — его `/today` показывает его собственные, как у любого менеджера.

### 2. Все блоки — `assignedToId = текущий пользователь`, независимо от `leadVisibility`

Глобальная настройка видимости (`ALL`/`OWN`) управляет тем, что менеджер видит в общем списке/доске. «Сегодня» — всегда «то, что назначено на меня», даже если компания настроена на `leadVisibility: ALL` (видеть в общем списке чужие лиды не означает, что они появятся в персональной сводке задач другого человека).

### 3. Без отдельного хранения — только запросы

Ни один блок не требует новой таблицы. Это композиция уже существующих запросов (`Lead` с фильтрами + `Task` с фильтрами + `lib/risk/computeRiskBatch.ts`).

## Блоки экрана

| Блок | Запрос (концептуально) |
| --- | --- |
| **Новые лиды** | `Lead`: `assignedToId = me`, `closeType IS NULL`, без `LEAD_TAKEN_IN_WORK`, `createdAt` за последние 24 часа |
| **Необработанные лиды** | `Lead`: `assignedToId = me`, `closeType IS NULL`, без `LEAD_TAKEN_IN_WORK` (без ограничения по времени — это постоянный «хвост», новые лиды — его горячая часть) |
| **Задачи на сегодня** | `Task`: `assignedToId = me`, `dueDate` сегодня, `status IN (TODO, IN_PROGRESS)` |
| **Просроченные задачи** | `Task`: `assignedToId = me`, `dueDate < now()`, `status IN (TODO, IN_PROGRESS)` |
| **Лиды без следующего действия** | `Lead`: `assignedToId = me`, `closeType IS NULL`, нет открытой задачи — пересекается с `risk = yellow` («Нет следующего шага») |
| **Лиды, скоро станут просроченными** | `Lead`: `assignedToId = me`, `risk.level = yellow` (через `computeRiskBatch`) |
| **Лиды, требующие вмешательства** | `Lead`: `assignedToId = me`, `risk.level = red` |

Блоки **не взаимоисключающие** — один лид может одновременно попасть в «необработанные» и в «требующие вмешательства». Это ожидаемо: экран не делит лиды по корзинам, а показывает разные срезы одного и того же рабочего хвоста.

### Пустые блоки

Если блок пуст — не показывается совсем (а не «0 элементов»), чтобы экран не выглядел загруженным, когда работы на самом деле нет.

### Персональные счётчики (над блоками)

Над семью блоками — строка из 4 карточек-счётчиков, та же визуальная форма, что была у старого компанийского дашборда (`StatCard`), но со счётом строго по текущему пользователю (`assignedToId = me`), не по всей компании:

| Счётчик | Запрос |
| --- | --- |
| Всего лидов | `Lead.count`: `assignedToId = me` (без ограничения по времени и статусу) |
| Новых сегодня | `Lead.count`: `assignedToId = me`, `createdAt` за текущий календарный день |
| В работе | `Lead.count`: `assignedToId = me`, `closeType IS NULL` |
| Сделок | `Lead.count`: `assignedToId = me`, `closeType = WON` |

Считается напрямую в `page.tsx` (4 независимых `count`), не через `GET /api/today` — это не один из 7 блоков, а отдельная лёгкая агрегация без риска/задач.

### Лиды — карточки в сетке, не построчный список

Лид-блоки («Новые», «Необработанные», «Требуют вмешательства», «Скоро станут просроченными», «Без следующего действия») рендерятся как сетка компактных карточек (`grid-cols-1 sm:grid-cols-2 xl:grid-cols-3`), не как построчный список — каждая карточка со своей рамкой/тенью, `RiskBadge` служит цветовой меткой. Блоки задач («Задачи на сегодня», «Просроченные задачи») остаются построчным списком — `TodaySection` принимает `layout: 'list' | 'grid'`.

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
| --- | --- | --- | --- | --- |
| GET | `/api/today` | Все блоки одним ответом | Session | Любой авторизованный (фильтр всегда «моё») |

Один эндпоинт, не семь — блоки достаточно лёгкие, чтобы не разносить по отдельным запросам и не множить round-trips на загрузке главного экрана.

### `GET /api/today`

Каждый блок ограничен 20 записями (сортировка «новее/срочнее — выше») и отдаёт `total` — полный размер до среза, чтобы UI мог предложить переход в `/leads`/`/pipeline` вместо неограниченного списка на одном экране.

**Response 200:**

```json
{
  "newLeads": { "items": [ { "id": "...", "name": "...", "risk": { "level": "yellow", "reason": "..." } } ], "total": 3 },
  "unprocessedLeads": { "items": [ ... ], "total": 12 },
  "tasksToday": { "items": [ ... ], "total": 4 },
  "overdueTasks": { "items": [ ... ], "total": 1 },
  "leadsWithoutNextAction": { "items": [ ... ], "total": 2 },
  "leadsApproachingDeadline": { "items": [ ... ], "total": 5 },
  "leadsAtRisk": { "items": [ ... ], "total": 1 }
}
```

## Файлы, которые создаются

```
app/
└── (app)/today/page.tsx
└── api/today/route.ts

components/
└── today/
    ├── TodayBoard.tsx           # Server Component: раскладка блоков
    ├── TodayStatsRow.tsx        # Личные счётчики (Всего/Новых сегодня/В работе/Сделок) поверх components/blocks/StatCard.tsx
    ├── TodaySection.tsx         # Переиспользуемый блок с заголовком + списком/сеткой (layout: 'list' | 'grid')
    ├── TodayLeadRow.tsx         # Карточка лида в сетке (риск как цветовая метка + быстрые действия из leads.md)
    └── TodayTaskRow.tsx         # Компактная строка задачи (название, срок, имя лида)
```

`TodayLeadRow.tsx` переиспользует `LeadRowQuickActions.tsx` из `.docs/modules/leads.md` — быстрые действия одинаковы везде, где показывается строка лида.

## Серверные правила безопасности

1. **`assignedToId = session.user.id` — жёстко, не параметр запроса.** Нельзя запросить «Сегодня» другого пользователя через query-параметр.
2. **`companyId` из сессии**, как везде.
3. **Один запрос на агрегацию данных, не семь** — `getTodayData.ts` собирает блоки за минимальное число обращений к БД (переиспользует `computeRiskBatch` из `risk.md`, который сам уже избегает N+1).

## Связи с другими модулями

- **`.docs/modules/risk.md`** — два блока («скоро просрочены», «требуют вмешательства») целиком производные от расчёта риска.
- **`.docs/modules/leads.md`** — переход по клику на карточку, быстрые действия в строке — переиспользуются, не дублируются.
- **`.docs/modules/tasks.md`** — блоки задач — обычные запросы к `Task` с фильтром по исполнителю.
- **`CLAUDE.md`** — `/today` — защищённый роут рабочей зоны, без отдельной логики в `proxy.ts` сверх общей.
