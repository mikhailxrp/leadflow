# Модуль: Задачи (tasks)

> Спецификация задач по лидам. Базовая механика не изменилась; добавилась вторая роль задачи — она же реализует «обязательное следующее действие» из Технички, без новой сущности в БД.
> Связанные файлы: `.docs/database.md` (`Task`, `TaskStatus`), `.docs/modules/leads.md` (отображение как «следующее действие»), `.docs/modules/risk.md` (риск при отсутствии открытой задачи), `.docs/modules/today.md`.

---

## Что не изменилось

CRUD задач, статусы (`TODO → IN_PROGRESS → DONE/CANCELLED`), `completedAt` только сервером, права (создание — ADMIN/MANAGER в пределах видимости, удаление — ADMIN only, редактирование — автор или ADMIN), каскад с лидом, навигация `TaskItem → /leads/:leadId` — без изменений. См. предыдущую версию документа.

---

## Что изменилось

### 1. Задача — это и «следующее действие»

Техничка вводит понятие «у каждого активного лида должен быть следующий шаг» (FR-170…172) как отдельное требование. Решение — **не заводить новую сущность**, а считать следующим действием обычную открытую задачу (`status IN (TODO, IN_PROGRESS)`) лида с самым ранним `dueDate`.

```
Самая ранняя открытая задача лида = «следующее действие», показывается в карточке/списке/«Сегодня»
Открытых задач нет = «нет следующего действия» — сигнал риска (.docs/modules/risk.md), не ошибка
```

Это не меняет ничего в самой модели `Task` — меняется только то, как её читают другие модули.

### 2. Готовые типы задач кнопками (нюанс UI, не схемы)

Техничка предлагает оформить типовые задачи (позвонить, написать, отправить документ, подготовить расчёт, назначить встречу, уточнить решение) готовыми кнопками при создании — это предзаполнение поля `title` в форме создания, не новое поле и не enum `TaskType`. `title` остаётся свободным текстом — кнопки только ускоряют типовой ввод.

### 3. Промпт при смене этапа

При смене этапа лида без открытой задачи — модалка «Что делаем дальше и когда?» (см. `.docs/modules/leads.md`). Создаёт обычную задачу через тот же `POST /api/leads/:id/tasks` — не отдельный эндпоинт.

### 4. Мутации — обычные проверки, без guard'а подписки

Создание/изменение/удаление задачи — обычные мутации с проверкой видимости и роли. Отдельного guard'а подписки не существует — биллинга в продукте нет. Единственная точка проверки доступа компании — `Company.isBlocked` при входе (`.docs/modules/auth.md`).

## Связи с другими модулями

- **`.docs/modules/leads.md`** — «следующее действие» в карточке/списке — это открытая задача лида; промпт при смене этапа.
- **`.docs/modules/risk.md`** — отсутствие открытой задачи — один из факторов жёлтого риска.
- **`.docs/modules/today.md`** — блоки «задачи на сегодня»/«просроченные» — обычные запросы к `Task`.
- **`.docs/modules/admin-users.md`** — список менеджеров для селекта исполнителя.
- **`.docs/database.md`** — модель `Task` без структурных изменений.
