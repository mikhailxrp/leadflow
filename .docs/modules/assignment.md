# Модуль: Распределение лидов (assignment)

> Спецификация назначения ответственного менеджера: ручное назначение админом и автораспределение round-robin. Переключение режима, переназначение, защита от гонок.

## Цели модуля

После завершения этого модуля:

- Новый лид может назначаться менеджеру автоматически (round-robin)
- Админ может назначить/переназначить ответственного вручную
- Режим распределения переключается в настройках (ручной / авто)
- Распределение устойчиво к гонкам при одновременном приёме лидов
- Каждое назначение пишется в историю лида
- Корректно обрабатываются краевые случаи (нет менеджеров, заблокированные)

**Не входит в модуль:**

- Приём лида → `.docs/modules/leads-intake.md` (вызывает `assignLead` после коммита)
- Уведомление назначенному менеджеру → `.docs/modules/notifications.md`
- UI переключателя режима → `.docs/modules/app-settings.md`
- Список/карточка лида → `.docs/modules/leads.md`

## Два режима распределения

Реализуются **оба** (FR-30, FR-31). Режим выбирает админ — `Company.settings.assignMode` (`MANUAL | ROUND_ROBIN`).

```
Если → assignMode === ROUND_ROBIN
То   → при приёме лида сервер сам назначает следующего менеджера по кругу

Если → assignMode === MANUAL
То   → лид приходит без ответственного (assignedToId = null)
       → админ назначает вручную в карточке лида
```

Переключение режима — в настройках (`.docs/modules/app-settings.md`). Дефолт — `MANUAL` (безопасный: лид не «потеряется» у занятого менеджера, админ распределит сам).

## Архитектурные решения

### 1. Единая точка входа `assignLead`

Вся логика назначения — в `lib/assignLead.ts`. Вызывается:

- автоматически после приёма лида (если режим `ROUND_ROBIN`)
- из API при ручном/переназначении

```typescript
// lib/assignLead.ts — сигнатуры
// Автораспределение: выбрать следующего и назначить
async function autoAssignLead(leadId: string, companyId: string): Promise<void>;
// Явное назначение конкретному менеджеру (ручное / переназначение)
async function assignLeadTo(leadId: string, managerId: string | null, actorId: string): Promise<void>;
```

### 2. Назначение — после коммита приёма

`leads-intake` сохраняет лид, и только после успешного коммита вызывает `autoAssignLead`. Если назначение упадёт — лид уже сохранён (не потерян), останется без ответственного, админ назначит вручную. Приём важнее назначения.

### 3. Round-robin через курсор + advisory lock

Курсор последнего назначенного менеджера — `Company.settings.roundRobinCursor`. Распределение оборачивается в PostgreSQL advisory lock на `companyId`, чтобы параллельные приёмы не назначили лиды одному менеджеру (см. `.docs/database.md` → «Критичные транзакции → Автораспределение»).

### 4. Назначение пишет событие

Любое назначение/переназначение → событие `ASSIGNED` с `payload: { fromUserId, toUserId }`. Источник истории и аудита.

## Ручное назначение

### Поведение (FR-30)

В карточке лида — селект «Ответственный». Админ выбирает менеджера → `PATCH /api/leads/:id/assign`.

```
Кто может назначать вручную:
- ADMIN — любому лиду любого менеджера
- MANAGER — НЕ может переназначать (только админ управляет ответственными)
```

> Примечание: по PRD назначение ответственных — право администратора (раздел «Роли»). Менеджер работает со своими лидами, но не перераспределяет их.

### Серверная логика

```typescript
await assignLeadTo(leadId, managerId, actorId);
// внутри:
// 1. проверить, что лид принадлежит компании актора
// 2. проверить, что managerId — активный менеджер той же компании (или null для снятия)
// 3. update Lead.assignedToId
// 4. event ASSIGNED { fromUserId, toUserId: managerId }
```

`managerId = null` снимает ответственного (лид становится нераспределённым).

## Автораспределение (round-robin)

### Поведение (FR-31)

При приёме лида в режиме `ROUND_ROBIN` сервер назначает **следующего активного менеджера по кругу**.

```
Менеджеры компании (активные, по дате создания): [A, B, C]
roundRobinCursor = B  (последний получивший)

Новый лид → следующий после B → C → cursor = C
Ещё лид   → следующий после C → A (по кругу) → cursor = A
```

### Транзакция

Полный код — `.docs/database.md` → «Критичные транзакции → Автораспределение (round-robin)». Логика:

```
1. advisory lock на companyId (сериализует параллельные распределения)
2. выбрать активных менеджеров (role=MANAGER, isBlocked=false) по createdAt
3. если менеджеров нет → выход, лид без ответственного
4. найти индекс курсора, взять следующего по модулю длины
5. update Lead.assignedToId + event ASSIGNED + сдвинуть курсор в settings
```

### Почему advisory lock

Без него два одновременно пришедших лида прочитают один `roundRobinCursor` и оба уйдут одному менеджеру — равномерность сломается. Lock сериализует распределение в пределах компании. Освобождается автоматически на COMMIT/ROLLBACK.

### Учитываются только активные менеджеры

Заблокированные (`isBlocked=true`) и админы в круг не входят. Только `role=MANAGER, isBlocked=false`.

## Переназначение

### Поведение (FR-33)

Админ меняет ответственного у лида, у которого он уже есть. Тот же эндпоинт, что ручное назначение (`PATCH /api/leads/:id/assign`).

Событие `ASSIGNED` пишет и старого, и нового: `payload: { fromUserId: oldManager, toUserId: newManager }`. В истории лида видно всю цепочку передач.

### Не трогает курсор round-robin

Ручное переназначение НЕ сдвигает `roundRobinCursor` — курсор только для автораспределения. Иначе ручная правка собьёт автоматический круг.

## Краевые случаи

```
Случай: режим ROUND_ROBIN, но активных менеджеров нет
→ лид сохраняется с assignedToId = null
→ событие ASSIGNED не пишется
→ админ увидит нераспределённый лид и назначит вручную

Случай: единственный менеджер заблокирован после назначения ему лидов
→ его лиды остаются за ним (история сохраняется)
→ новые лиды (round-robin) ему не идут (он не активен)
→ переназначение его лидов — вручную админом

Случай: менеджер, на котором стоит roundRobinCursor, удалён/заблокирован
→ при следующем распределении его id не найдётся в списке активных
→ findIndex вернёт -1 → берётся первый активный (next = managers[0])
→ круг продолжается корректно

Случай: ручное назначение менеджеру другой компании (в SaaS-будущем)
→ проверка companyId менеджера === companyId лида → иначе 400
```

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
| --- | --- | --- | --- | --- |
| PATCH | `/api/leads/:id/assign` | Назначить / переназначить / снять ответственного | Session | ADMIN only |

> Автораспределение не имеет своего эндпоинта — оно вызывается сервером изнутри приёма лида (`autoAssignLead`).

### `PATCH /api/leads/:id/assign`

**Request (назначить):** `{ "managerId": "clxxx" }`
**Request (снять ответственного):** `{ "managerId": null }`

**Response 200:**

```json
{ "success": true, "assignedTo": { "id": "clxxx", "name": "Алексей" } }
```

**Response 400:**

```json
{ "success": false, "error": "MANAGER_NOT_FOUND" | "MANAGER_BLOCKED" | "WRONG_COMPANY" }
```

**Response 403 (не админ):**

```json
{ "success": false, "error": "FORBIDDEN" }
```

## Серверные правила безопасности

1. **Все операции — с проверкой `companyId`.** Лид и назначаемый менеджер — одной компании. Назначение менеджера чужой компании → `400` (важно для SaaS-будущего).

2. **Назначение вручную — только ADMIN.** Проверка роли на сервере. Менеджер не перераспределяет лиды.

3. **Round-robin учитывает только активных менеджеров** (`role=MANAGER, isBlocked=false`). Заблокированные и админы исключены.

4. **Advisory lock обязателен в автораспределении.** Без него гонка при одновременном приёме сломает равномерность круга.

5. **Курсор сдвигает только автораспределение.** Ручное назначение/переназначение не трогает `roundRobinCursor`.

6. **Назначение пишет событие** `ASSIGNED` через `lib/events.ts` с `fromUserId` и `toUserId`. Источник истории передач.

7. **Назначение не блокирует приём.** Вызывается после коммита приёма; его сбой оставляет лид нераспределённым, но сохранённым.

8. **`managerId` валидируется** — существует, активен, той же компании. `null` разрешён (снять ответственного).
