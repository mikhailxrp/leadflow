## 1. Обзор продукта

Лид-Канал — CRM для малого бизнеса, который ведёт продажи в Excel/Google Таблицах, блокнотах и чатах и пока не готов внедрять тяжёлую CRM (Битрикс24, amoCRM).

### 1.1 Главный принцип

> **Лид-Канал должен быть системой, в которой невозможно незаметно потерять входящий лид.**

```
Лид поступил → система сохранила → определила источник → назначила ответственного →
запустила таймер первого ответа → уведомила менеджера → менеджер взял лид в работу →
зафиксировал результат → назначил следующий шаг → система контролирует срок →
при просрочке напоминает менеджеру → при дальнейшем бездействии уведомляет руководителя →
лид закрывается сделкой или понятной причиной отказа
```

Всё, что не поддерживает этот сценарий, не попадает в продукт без отдельного подтверждённого спроса (см. раздел 9).

### 1.2 Проблема и решение

Малый бизнес теряет заявки: они приходят из разных каналов, нигде не собираются вместе, менеджеры не видят полной картины, никто не контролирует, не завис ли лид без ответа. Лид-Канал — единая точка приёма, автоматический контроль сроков реакции, видимый риск потери, понятные причины отказа и минимальная отчётность о том, не утекают ли заявки.

### 1.3 Цели

- Принимать лиды из любых источников без потерь, включая ручной ввод и импорт исторических данных
- Не дать лиду остаться без ответственного, без первого ответа в срок и без следующего шага
- Сделать риск потери лида видимым раньше, чем это заметит клиент
- Дать руководителю и собственнику ответ на три вопроса: теряем ли мы лиды, кто и где их теряет, какие источники дают результат
- Расти как управляемая многотенантная платформа: новые компании подключает платформенный администратор, без архитектурного долга на масштаб

### 1.4 Метрики успеха


| Метрика                                               | Целевое значение                                                                             |
| ----------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| Потеря лидов                                          | 0% — каждая заявка сохранена, независимо от структуры данных, дублей или блокировки компании |
| Время приёма лида                                     | < 2 сек от webhook до отображения                                                            |
| Доставка уведомления                                  | < 5 сек (интерфейс и Telegram)                                                               |
| Адаптация к новым полям формы                         | Без участия разработчика                                                                     |
| Видимость риска                                       | Зависший/просроченный лид виден в «Сегодня» и в «Контроле» без отдельного поиска             |
| Время от создания компании до первого рабочего экрана | Одна ссылка-приглашение — клиент сам задаёт пароль и сразу видит интерфейс                   |

## 2. Платформа и компании

### 2.1 Модель продукта

Лид-Канал — **управляемая многотенантная платформа**, не self-serve SaaS. Компании не регистрируются сами — их создаёт платформенный администратор. Биллинга внутри продукта нет: коммерческие договорённости с клиентом находятся вне системы.


| Аспект                   | Решение                                                                            |
| ------------------------ | ---------------------------------------------------------------------------------- |
| Появление новой компании | Создаёт платформенный администратор, не сама компания                              |
| Доступ клиента           | По ссылке-приглашению — клиент сам задаёт пароль                                   |
| Биллинг                  | Отсутствует в продукте                                                             |
| Архитектура              | Мультитенантная — много компаний на одной инсталляции                              |
| Поддержка                | Платформенный администратор может войти внутрь компании (impersonation) для помощи |


### 2.2 Платформенный администратор

Отдельная от компаний сущность (`PlatformAdmin`, не `User`) — не принадлежит ни одной компании, не работает с лидами, не получает уведомлений о них.

**(v4.1)** Платформенных ролей две (`PlatformRole`): `SUPER_ADMIN` — платформенный администратор (этот раздел) и `MARKETER` — маркетолог (раздел 2.7). Роли — **не иерархия**, а два разных скоупа. «Свои» компании суперадмина — платформенные: созданные любым суперадмином или существовавшие до v4.1.


| ID      | Требование                                                                                                                                       |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| PLAT-01 | Первая запись создаётся одноразовым bootstrap-скриптом при деплое, не публичной регистрацией                                                     |
| PLAT-02 | Существующий платформенный администратор может создать ещё одного — их может быть больше одного                                                  |
| PLAT-03 | Создаёт компании, указывая название и email первого администратора                                                                               |
| PLAT-04 | Видит список всех компаний; у компаний, созданных маркетологами, скрыты `companyId` и кнопки управления/входа — они видны только как сигнал «живая/мёртвая» |
| PLAT-05 | Видит отчёт об активности каждой компании (в рамках видимости): последний вход, число лидов за период, число активных пользователей — **не** лиды как рабочий процесс; плюс активность маркетологов |
| PLAT-06 | Блокирует/разблокирует свои (платформенные) компании без удаления данных                                                                         |
| PLAT-07 | Входит «как» любой пользователь своей компании (impersonation); в компанию маркетолога — только по вручную предоставленному `companyId`          |
| PLAT-08 | Никогда не знает и не задаёт пароль пользователя компании                                                                                        |
| PLAT-09 | Ведёт дату продления подписки своих компаний (годовой офлайн-договор) и получает автоуведомление (email-дайджест) о приближении/просрочке срока — без биллинга внутри продукта |
| PLAT-10 | Создаёт и блокирует маркетологов; блокировка маркетолога каскадно блокирует его компании (раздел 2.8)                                            |
| PLAT-11 | Может предоставить выбранному маркетологу доступ к своей (платформенной) компании — грант; грант не передаёт управление (блокировку, дату платежа) |
| PLAT-12 | Видит страницу логов: журнал событий по фильтрам (компания обязательна), изначально показываются только фильтры (раздел 2.9)                     |


### 2.3 Создание компании и приглашение администратора


| ID      | Требование                                                                                                                    |
| ------- | ----------------------------------------------------------------------------------------------------------------------------- |
| INIT-01 | Платформенный администратор указывает название компании и email будущего администратора                                       |
| INIT-02 | Транзакция создаёт компанию, 5 дефолтных этапов воронки, дефолтный набор причин отказа, приглашение со сроком действия 7 дней |
| INIT-03 | Клиент по ссылке-приглашению сам задаёт пароль и имя — становится администратором компании                                    |
| INIT-04 | Повторное использование уже принятого или просроченного приглашения — отклоняется                                             |
| INIT-05 | Email пользователя уникален глобально, как и раньше                                                                           |


### 2.4 Блокировка компании


| ID       | Требование                                                                                                       |
| -------- | ---------------------------------------------------------------------------------------------------------------- |
| BLOCK-01 | Платформенный администратор блокирует компанию вручную, без удаления данных                                      |
| BLOCK-02 | Заблокированная компания не может войти ни одним пользователем                                                   |
| BLOCK-03 | Приём лидов по вебхукам продолжает работать у заблокированной компании — лид нельзя потерять даже при блокировке |
| BLOCK-04 | Разблокировка восстанавливает доступ со следующего входа                                                         |


### 2.5 Вход от имени компании (impersonation)


| ID     | Требование                                                                                                               |
| ------ | ------------------------------------------------------------------------------------------------------------------------ |
| IMP-01 | Платформенный администратор выбирает компанию и конкретного пользователя, от имени которого войти                        |
| IMP-02 | Сессия создаётся как у выбранного пользователя — те же права, что и у него                                               |
| IMP-03 | Интерфейс показывает постоянный баннер о режиме поддержки и кнопку выхода из него                                        |
| IMP-04 | Начало и конец impersonation-сессии фиксируются в журнале                                                                |
| IMP-05 | Каждое действие, совершённое во время impersonation, помечено в истории как выполненное поддержкой от имени пользователя |


### 2.6 Срок подписки компании (продление по офлайн-договору)

Компания подключается на годовой период по офлайн-договору; биллинга внутри продукта нет (раздел 9). Платформенный администратор ведёт дату продления как напоминание — не тариф, не оплата.


| ID     | Требование                                                                                                        |
| ------ | ----------------------------------------------------------------------------------------------------------------- |
| SUB-01 | У компании есть дата следующего платежа (`nextPaymentAt`); при создании по умолчанию +1 год                        |
| SUB-02 | Платформенный администратор может установить/изменить/сбросить дату                                                |
| SUB-03 | За 2 недели до даты и при просрочке компания выделяется красным в списке и карточке на платформенном уровне        |
| SUB-04 | Email-дайджест о компаниях с приближающимся/просроченным сроком (раз в сутки) уходит **владельцу**: по платформенным компаниям — каждому активному суперадмину, по компаниям маркетолога — этому маркетологу (v4.1) |
| SUB-05 | Просрочка не блокирует вход и не останавливает приём лидов — блокировка остаётся отдельным ручным действием        |
| SUB-06 | Это не биллинг: нет процессинга, тарифов, фичагейтов, самооплаты                                                   |


### 2.7 Маркетолог (v4.1)

Урезанная платформенная роль (`PlatformAdmin` с `role: MARKETER`). Создаётся платформенным администратором, отвечает за созданные им компании и ведёт квалификацию их лидов для обучения рекламных систем. Полная спецификация — `.docs/modules/platform-marketer.md`.


| ID     | Требование                                                                                                                                     |
| ------ | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| MKT-01 | Создаётся и блокируется только платформенным администратором; входит через тот же `/platform/login` (сессия `kind: "platform"`, `role: MARKETER`) |
| MKT-02 | Создаёт компании тем же флоу, что и суперадмин (раздел 2.3), становится их владельцем                                                          |
| MKT-03 | Видит **только свои** компании (созданные + предоставленные грантом), включая их `companyId`                                                   |
| MKT-04 | Управляет своими компаниями тем же набором, что и суперадмин своими: блокировка/разблокировка, дата платежа, активность                        |
| MKT-05 | Входит внутрь своих компаний в ограниченном режиме — **не** impersonation реального пользователя, а виртуальный доступ с явным allow-list      |
| MKT-06 | Внутри компании: лиды — просмотр и квалификация (без удаления, «взять в работу», назначения ответственного, закрытия, создания); воронка — только просмотр; отчёты; интеграции |
| MKT-07 | Не управляет платформенными пользователями (ни администраторами, ни маркетологами) и не выдаёт гранты                                          |
| MKT-08 | Видит логи своих компаний с фильтрами по компаниям и режимом «путь лида» (фильтр по лидам, таймлайн событий лида)                              |
| MKT-09 | Вход и выход маркетолога внутрь компании логируются (`MARKETER_ACCESS_STARTED`/`ENDED`), каждое его действие аннотировано в истории             |

### 2.8 Каскадная блокировка маркетолога (v4.1)

Маркетолог сам отвечает за работоспособность системы для созданных им компаний.


| ID      | Требование                                                                                                                        |
| ------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| CASC-01 | Блокировка маркетолога блокирует все его активные компании в одной транзакции (с пометкой каскада)                                 |
| CASC-02 | Платформенные администраторы получают email со списком заблокированных компаний и контактами их администраторов (имя, email)       |
| CASC-03 | Приём лидов по вебхукам заблокированных компаний продолжается — «лид нельзя потерять» действует без исключений                     |
| CASC-04 | Разблокировка маркетолога разблокирует только компании, заблокированные каскадом; заблокированные им вручную ранее остаются заблокированными |
| CASC-05 | Ручная блокировка/разблокировка компании владельцем сбрасывает пометку каскада                                                     |
| CASC-06 | Компании, доступные маркетологу по гранту, каскадом не затрагиваются (они принадлежат суперадмину)                                 |

### 2.9 Логи платформенного уровня (v4.1)


| ID     | Требование                                                                                                                   |
| ------ | ------------------------------------------------------------------------------------------------------------------------------ |
| LOG-01 | Страница `/platform/logs` доступна обеим платформенным ролям в рамках их видимости компаний                                    |
| LOG-02 | Изначально показываются только фильтры; выборка — после выбора компании (фильтр по компании обязателен)                        |

| LOG-05 | Источник данных — существующий журнал `events`, новых таблиц нет                                                               |

## 3. Роли внутри компании

Три роли — **иерархия**, не три изолированных набора прав: у Администратора есть всё, что есть у Руководителя, у Руководителя — всё, что есть у Менеджера.

### 3.1 Менеджер


| Возможность                                              | Доступ                                         |
| -------------------------------------------------------- | ---------------------------------------------- |
| Просмотр лидов                                           | Только свои (жёсткое правило, не настраивается) |
| «Взял в работу», следующее действие, закрытие лида       | Да                                             |
| Свои задачи, свои просрочки                              | Да                                             |
| Доступная история                                        | Да                                             |
| Управление пользователями, настройками, отчёты, контроль | Нет                                            |


### 3.2 Руководитель

Всё, что доступно Менеджеру, плюс:


| Возможность                                                    | Доступ                                                                             |
| -------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| Просмотр всех лидов компании                                   | Да, всегда — не зависит от настройки видимости (та применяется только к Менеджеру) |
| Все сотрудники                                                 | Да                                                                                 |
| Контроль (счётчик активности менеджеров)                       | Да                                                                                 |
| Отчёты                                                         | Да                                                                                 |
| Распределение нагрузки (ручное (пере)назначение лида)          | Да                                                                                 |
| Настройки, пользователи, интеграции, причины отказа, нормативы | Нет — это уровень Администратора                                                   |


### 3.3 Администратор

Всё, что доступно Руководителю, плюс:


| Возможность                                                                  | Доступ |
| ---------------------------------------------------------------------------- | ------ |
| Настройки системы (видимость, режим распределения, нормативы, рабочее время) | Да     |
| Пользователи (создание, блокировка, роли)                                    | Да     |
| Дополнительные поля, этапы воронки                                           | Да     |
| Источники (интеграции, API-ключи)                                            | Да     |
| Правила автоматического распределения                                        | Да     |
| Причины отказа (справочник)                                                  | Да     |
| Удаление лидов                                                               | Да     |

### 4.2 Защита от дублей


| ID     | Требование                                                                                                                              |
| ------ | --------------------------------------------------------------------------------------------------------------------------------------- |
| FR-100 | Совпадение по телефону или email с существующим лидом той же компании — потенциальный дубль                                             |

| FR-102 | При ручном создании — предупреждение с превью совпадения **до** сохранения, с возможностью «всё равно создать» или открыть существующий |
| FR-103 | При вебхук-приёме — лид создаётся всегда, пометка ставится **после**, асинхронно                                                        |
| FR-104 | Пометка дубля видна в карточке лида и в списке                                                                                          |
| FR-105 | Слияние записей не реализуется                                                                                                          |

## 8. Нефункциональные требования

### 8.1 Безопасность

- HTTPS, bcrypt, хэшированные API-ключи, валидация и санитизация, журналирование — без изменений
- **Платформенная сессия и сессия компании никогда не смешиваются** — разный тип сессии, не просто разная роль
- **Пароль клиента никогда не известен платформенному администратору** — доступ внутрь компании только через impersonation, с обязательным журналированием начала/конца и аннотацией действий
- **(v4.1)** Платформенные роли (`SUPER_ADMIN`/`MARKETER`) — не иерархия; каждый платформенный эндпоинт объявляет явный список допустимых ролей
- **(v4.1)** Права маркетолога внутри компании — явный allow-list с deny-by-default; маркетолог никогда не impersonate'ит реального пользователя
- **(v4.1)** Скрытие `companyId` компаний маркетологов от суперадмина — граница ответственности в UI, не гарантия конфиденциальности (`companyId` присутствует в URL вебхуков); настоящая защита — журналирование каждого входа

### 8.2 Производительность, 8.3 Адаптивность и локализация — без изменений

### 8.4 Масштабируемость

- Мультитенантность — рабочий режим с первого дня
- Новые источники добавляются как строковое значение `source`, без миграций
- Новые роли — добавлением значения `UserRole` и обновлением `ROLE_RANK`, без переписывания проверок доступа по всему коду
- Event Sourcing остаётся основой для истории и отчётов

| Злоупотребление impersonation                      | Низкое (платформенных админов мало и они доверенные) | Журналирование начала/конца + аннотация действий — для разбора постфактум, не превентивный технический барьер |

## 11. Ключевые архитектурные принципы

1. Мультитенантность с первого дня
2. Лид нельзя потерять — ни по структуре данных, ни по дублю, ни по блокировке компании
3. Настраиваемая воронка — данные, не код
4. Event Sourcing — `events` как единственный источник истории
5. Серверная защита данных по `companyId` на каждом запросе
6. Распределение — конфигурация, не зашитая в код логика
7. Настройки отдельно от применения
8. Уведомления не блокируют приём
9. Ключи и пароли — только хэши
10. **Дубль — сигнал для человека, не решение системы.** Пометка-связка, не слияние
11. **Риск вычисляется, а не хранится** — производное от `events`/`Lead`/`Task` на момент запроса
12. **Роли — иерархия, проверяемая функцией, не перечислением значений enum** — `hasMinRole()`, не `role === "ADMIN" || role === "HEAD"`
13. **Платформенный уровень не редактирует данные компаний напрямую** — только через обычный прикладной код от имени реального пользователя (impersonation), никогда через отдельный «бэкдор»-путь

## 13. Поддержка

Без биллинга поддержка не привязана к статусу подписки — она привязана к самому факту работы с платформой.


| Аспект                                                                | Решение                                                                                                                                                                   |
| --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Поддержка                                                             | Платформенный администратор может войти внутрь компании (impersonation) и помочь напрямую — это и есть основной канал поддержки «руками»                                  |
| Доработки сверх продукта                                              | Отдельная договорённость, вне рамок этого документа                                                                                                                       |
| Резервное копирование                                                 | На стороне платформы, расписание и сроки хранения документируются отдельно                                                                                                |
| Экспорт данных                                                        | Доступен в любой момент, независимо от статуса компании (включая заблокированную — блокировка касается входа, не экспорта по запросу через платформенного администратора) |
| Удаление данных по запросу                                            | Поддерживается, политика хранения документируется отдельно                                                                                                                |
| Изменения сторонних сервисов (хостинг, Telegram Bot API, API Яндекса) | Вне зоны ответственности продукта                                                                                                                                         |

## 14. Роадмап

> Главное отличие от предыдущей версии: убрана целая фаза (биллинг), вместо самостоятельной регистрации компаний — отдельная ранняя фаза платформенного администратора, без которой не появится ни одна компания для тестирования остального.

### Фаза 0 — Фундамент

Каркас Next.js 16 + Prisma + PostgreSQL. Полная схема (`.docs/database.md`), включая `PlatformAdmin`, `CompanyInvite`, роль `HEAD`. `proxy.ts` редиректит неавторизованных по трём типам сессии (платформа/компания/публичные пути). Health-check.

### Фаза 1 — Мультитенантная инфраструктура и CI/CD (ручная настройка сервера)

VPS, Nginx, PM2, self-hosted GitHub Actions Runner — вручную, один раз. Ветки `dev`/`main` → отдельные процессы на портах 3001/3000, отдельные домены, отдельные базы Postgres.

### Фаза 2 — Платформенный администратор

Bootstrap-скрипт создаёт первую запись `PlatformAdmin` из ENV. `/platform/login`, отдельная сессия (`session.kind: "platform"`). Создание компании + приглашение (транзакция из `database.md`). Список компаний, блокировка/разблокировка. Создание дополнительных платформенных администраторов. Без этой фазы ни одна компания не появится для дальнейшей разработки и тестирования.

### Фаза 3 — Авторизация компании

`/accept-invite` — приём приглашения, установка пароля, автоматический вход. `/login`, NextAuth v5, роль (`MANAGER`/`HEAD`/`ADMIN`) в сессии. `/forgot-password`/`/reset-password`. `proxy.ts` защищает рабочую зону по `hasMinRole`.

### Фаза 4 — Каркас рабочей зоны + сайдбар + дашборд

Навигация показывает только то, что доступно роли (например, пункты «Контроль»/«Отчёты» не отображаются Менеджеру).

### Фаза 5 — Приём лидов (ядро) + защита от дублей

`lib/intake/` + `flagPossibleDuplicates()`. Webhook-эндпоинты Tilda/WordPress/универсальный. Ручное создание с синхронной проверкой дублей. WhatsApp не реализуется ни в каком виде.

### Фаза 6 — Список лидов + поиск + фильтры + риск-индикатор

Таблица, поиск, фильтры, колонка риска, быстрые действия из строки.

### Фаза 7 — Карточка лида + «Взял в работу» + закрытие

«Взял в работу», блок риска, «Закрыть сделкой»/«Закрыть отказом», пометка дубля.

### Фаза 7.5 — Причины отказа

### Фаза 8 — Админка: настройка воронки

### Фаза 10 — Админка: пользователи

### Фаза 23 — Финальное тестирование и сдача

Чек-лист для прода: ENV без платёжных секретов (их больше нет), bootstrap-переменные платформенного администратора убраны из `.env` после первого запуска, резервное копирование настроено, HTTPS и rate limit активны на обоих окружениях.
