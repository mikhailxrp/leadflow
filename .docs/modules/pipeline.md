## Цели модуля

После завершения этого модуля:

- Лиды отображаются на Kanban-доске по этапам
- Карточку можно перетащить между этапами, дата фиксируется автоматически
- Каждый этап может иметь свой лимит допустимого времени — зависшие карточки отмечены визуально прямо на доске
- Колонка показывает счётчик лидов и среднее время нахождения
- Доска фильтруется по ответственному
- Закрытые лиды по умолчанию не загромождают рабочую доску
- Админ управляет этапами (создание/переименование/удаление/порядок) — без изменений в механике

**Не входит в модуль:**

- Приём лидов → `.docs/modules/leads-intake.md`
- Список/карточка лида → `.docs/modules/leads.md`
- Назначение менеджера → `.docs/modules/assignment.md`
- Алгоритм цвета риска (используется для отметки зависших карточек) → `.docs/modules/risk.md`

## Архитектурные решения

### 1–5 — без изменений

Этапы — данные (`PipelineStage`), не код. `Lead.stageId` — FK, не enum. Порядок — целое поле `order`. Дата смены статуса — из события `STAGE_CHANGED`, не отдельная колонка. Drag-and-drop — `@dnd-kit/core`, оптимистичное обновление с откатом при ошибке.

### 6. Лимит времени — per-stage, с фоллбэком на компанийский дефолт

```
У этапа задан stageTimeLimitDays → используется он
У этапа stageTimeLimitDays = null → используется Company.settings.stageStuckDaysDefault
```

Не два независимых механизма — один с переопределением. Старый компанийский дефолт (`stageStuckDaysDefault`, бывший `stageStuckDays`) остаётся базовым значением для этапов, у которых нет своего лимита.

### 7. Зависание считается тем же алгоритмом, что и общий риск

Визуальная отметка на карточке Kanban — это частный случай индикатора риска (`.docs/modules/risk.md`), не отдельная независимая проверка. Один источник правды для «зависло» — везде, где это показывается (доска, список, карточка, «Сегодня», «Контроль»).

## Kanban-доска

### Отображение — расширено

Страница `/pipeline`. Колонка: цветная полоска (`stage.color`), название, **счётчик лидов**, **среднее время нахождения** (FR-122).

```
Среднее время нахождения = avg(now() - дата последнего STAGE_CHANGED на этот этап)
                            по всем лидам, сейчас находящимся в колонке
```

Считается на чтении (не хранится), как и риск.

### Визуальная отметка зависших карточек (FR-121)

Карточка лида в колонке получает рамку/бейдж риска (тот же цвет, что в списке) прямо на доске — не нужно открывать карточку или список, чтобы увидеть, что лид завис.

### Фильтр по ответственному (FR-123)

Над доской — селект менеджера, фильтрует видимые карточки без перезагрузки страницы (клиентский фильтр над уже загруженными данными, либо серверный re-fetch — решение реализации).

### Источник данных — расширено

`visibilityWhere` — без изменений. **Доска по умолчанию показывает только открытые лиды** (`closeType = null`) — см. ниже.

## Drag-and-drop карточек — без изменений в механике

Транзакция смены этапа (`STAGE_CHANGED`) не менялась. Кто может двигать — без изменений (ADMIN любые, MANAGER в пределах видимости).

## Закрытые лиды на доске

### Поведение

Закрытие лида (сделка/отказ, см. `.docs/modules/leads.md`) — действие, не привязанное к этапу. Чтобы доска оставалась рабочим инструментом, а не архивом:

```
Доска по умолчанию: where closeType IS NULL (только открытые лиды)
Переключатель «Показать закрытые» — добавляет WON/LOST карточки серым, без drag-and-drop (закрытый лид не перетаскивается)
```

**Почему так, а не отдельная колонка «Закрыто»:** воронка — это про текущую работу. Закрытые лиды релевантны для отчётов (`.docs/modules/reports.md`) и истории, не для ежедневной Kanban-работы. Это решение, не зафиксированное явно в исходных требованиях — если нужно другое поведение (например, отдельная всегда видимая колонка «Сделка»/«Отказ»), это небольшая правка интерфейса, не архитектуры.

## Настройка этапов (админ)

### Создание, переименование — без изменений

`POST /api/stages`, `PATCH /api/stages/:id`. Добавлено поле `stageTimeLimitDays` (опционально) в обе формы.

### Минимум этапов — без изменений

Нельзя удалить последний этап.

## Лимит времени на этапе

### UI

В настройках этапа — необязательное числовое поле «Допустимое время на этапе (дней)». Пустое — используется компанийский дефолт.

```
PATCH /api/stages/:id { stageTimeLimitDays: 10 }
PATCH /api/stages/:id { stageTimeLimitDays: null }  // вернуть к дефолту компании
```

Используется и для визуальной отметки на доске, и для ежедневной сводки зависших лидов (`.docs/modules/notifications.md`).

## API-эндпоинты

| Метод | Путь | Назначение | Auth | Право |
| --- | --- | --- | --- | --- |
| GET | `/api/stages` | Список этапов | Session | ADMIN / MANAGER |
| POST | `/api/stages` | Создать (+ опционально `stageTimeLimitDays`) | Session | ADMIN only |
| PATCH | `/api/stages/:id` | Переименовать / цвет / `stageTimeLimitDays` | Session | ADMIN only |

## Серверные правила безопасности

1–8 без изменений относительно предыдущей версии (`companyId` везде, управление этапами только ADMIN, видимость при смене этапа, целевой этап той же компании, нельзя удалить последний этап, `moveTo` обязателен, переупорядочивание транзакционно, смена этапа пишет событие).

9. **`includeClosed` на `/api/pipeline/board` не снимает фильтр видимости** — только добавляет закрытые лиды в пределах того же `visibilityWhere`.

10. **Закрытые лиды не участвуют в drag-and-drop.** Сервер отклоняет смену `stageId` у лида с непустым `closeType` (`400 LEAD_CLOSED`) — если нужно его «переоткрыть», это отдельное действие (сейчас out of scope, см. `.docs/modules/leads.md`).
