## Цели модуля

После завершения этого модуля:

- Список и карточка лида показывают индикатор риска с причиной
- Менеджер фиксирует первый ответ однозначной кнопкой «Взял в работу»
- У активного лида видно следующее действие (или его отсутствие — это риск)
- Лид закрывается сделкой или отказом (отказ — только с причиной), независимо от текущего этапа воронки
- Возможные дубли видны и в списке, и в карточке
- Быстрые действия выполняются прямо из строки списка, без открытия карточки
- Работают поиск, фильтры, комментарии, история, редактирование, удаление — без изменений в механике
- Админ видит счётчик активности менеджеров

**Не входит в модуль:**

- Приём лидов → `.docs/modules/leads-intake.md`
- Алгоритм вычисления риска → `.docs/modules/risk.md` (здесь — только отображение)
- Kanban-доска → `.docs/modules/pipeline.md`
- Логика автораспределения → `.docs/modules/assignment.md`
- Создание/редактирование задач как таких → `.docs/modules/tasks.md` (здесь — только «следующее действие» как частный случай задачи)

## Видимость лидов

**Жёсткое правило без исключений и без настройки:** `MANAGER` видит и работает только со своими лидами (`assignedToId = session.user.id`) — список, карточка, воронка, задачи, напоминания, комментарии, смена этапа, взятие в работу, закрытие. `HEAD` и `ADMIN` видят все лиды компании всегда. Не настраивается администратором — никакого поля в `Company.settings` для этого нет (см. `lib/leads/visibilityFilter.ts`):

```typescript
function visibilityWhere(role: UserRole, userId: string) {
  if (hasMinRole(role, "HEAD")) return {}; // без ограничения
  return { assignedToId: userId };
}
```

Применяется в каждом запросе списка/карточки/подресурсов лида (задачи, напоминания, комментарии).

## Список лидов

### Отображение — расширено

Колонки: Клиент, Телефон, Источник, Ответственный, Этап, **Риск**, Время без обработки, Последнее действие, **Следующее действие**, Срок.

- **Риск** — цветной бейдж (зелёный/жёлтый/красный/серый) с подсказкой-причиной при наведении/тапе (см. `.docs/modules/risk.md`)
- **Следующее действие** — название и срок самой ранней открытой задачи лида; если открытых задач нет — «Нет следующего действия» (визуально как предупреждение, не ошибка)
- Источник — бейдж (см. `design-system.md` за иконками по источникам)
- Строка с пометкой дубля — небольшая иконка-ссылка на связанный лид

## Быстрые действия из списка

### Поведение (FR-141)

Каждая строка — набор инлайн-действий, без перехода в карточку:

| Действие | Механика |
| --- | --- |
| Назначить ответственного | Селект менеджера → `PATCH /api/leads/:id/assign` (без изменений, см. `assignment.md`) |
| Изменить этап | Селект этапа → `PATCH /api/leads/:id` (без изменений, см. `pipeline.md`) |
| Поставить следующее действие | Открывает компактную модалку создания задачи (переиспользует `AddTaskModal` из `tasks.md`) — без перехода на карточку |
| Изменить срок | Если у лида уже есть открытая задача — редактирует её `dueDate` напрямую; если нет — предлагает сначала поставить следующее действие |
| Открыть карточку | Переход на `/leads/:id` |
| Закрыть лид | Меню «Закрыть сделкой» / «Закрыть отказом» (отказ — модалка с обязательным выбором причины) |

**Зачем это нужно:** Техничка прямо формулирует — «чем меньше кликов требуется менеджеру, тем выше вероятность, что система будет использоваться». Все действия, которые раньше требовали открытия карточки, теперь доступны и из списка.

## Карточка лида

### Структура — обновлена

**Верхняя часть:** клиент, контакты, этап, ответственный, источник, **риск с причиной**, **следующее действие**.

**Основная часть:** комментарии, история, задачи, напоминания, маркетинговые данные, дополнительные поля.

**Быстрые кнопки:** **«Взял в работу»**, добавить комментарий, поставить задачу, назначить следующий шаг, сменить этап, **«Закрыть сделкой»**, **«Закрыть отказом»**.

### Фиксация открытия — без изменений

`GET /api/leads/:id` пишет `LEAD_OPENED` один раз на пользователя. **Важно: это не считается первым ответом** (см. ниже) — только просмотром.

---

## «Взял в работу»

### Поведение (FR-111)

Раньше первым ответом считалось любое из: открытие карточки, смена этапа, комментарий. Техничка прямо просит убрать эту двусмысленность — теперь первый ответ фиксируется **только** явным действием.

```
POST /api/leads/:id/take
  → проверка видимости (visibilityWhere)
  → если уже зафиксировано (есть LEAD_TAKEN_IN_WORK по этому лиду) → 400 ALREADY_TAKEN (повторный клик не дублирует)
  → событие LEAD_TAKEN_IN_WORK
  → это событие останавливает эскалацию (.docs/modules/notifications.md) — дальше ступени 1/2/3 не идут
```

Кнопка после нажатия превращается в неактивную метку «Взято в работу {время}». Видна тому, кто взял, и админу.

## Следующее действие

### Поведение (FR-112, FR-170…172)

«Следующее действие» — это **не новое поле**, а обычная открытая задача (`Task`, статус `TODO`/`IN_PROGRESS`) этого лида с самым ранним `dueDate`. Решение принято осознанно: не плодить параллельную сущность рядом с уже существующими задачами.

```
У лида есть открытая задача с ближайшим dueDate → она и показывается как «следующее действие»
У лида нет ни одной открытой задачи            → «Нет следующего действия» → сигнал риска (risk.md)
```

### Промпт при смене этапа (FR-171)

```
Менеджер меняет этап лида (drag-and-drop или из списка)
  → если у лида нет открытой задачи → модалка «Что делаем дальше и когда?»
  → менеджер может создать задачу сразу или закрыть модалку без действия
  → переход этапа выполняется в любом случае — это НЕ блокирующая проверка
```

## Закрытие лида (сделка/отказ)

### Поведение (FR-113)

Кнопки «Закрыть сделкой» / «Закрыть отказом» — действие, не привязанное к конкретному этапу воронки (см. `database.md` → `Lead.closeType`). Закрыть можно с любого этапа.

```
POST /api/leads/:id/close { closeType: "WON" }
  → Lead.closeType = WON, closedAt = now()
  → событие LEAD_WON

POST /api/leads/:id/close { closeType: "LOST", lossReasonId: "..." }
  → lossReasonId обязателен → без него 400 LOSS_REASON_REQUIRED
  → Lead.closeType = LOST, closedAt = now(), lossReasonId
  → событие LEAD_LOST
```

Закрытый лид остаётся видимым в списке (серый бейдж риска), но по умолчанию скрыт с Kanban-доски (см. `.docs/modules/pipeline.md`).

### Переоткрытие — out of scope MVP

Закрытие сейчас одностороннее (нет кнопки «вернуть в работу»). Если понадобится — отдельная небольшая доработка, не блокирует текущий объём.

## Возможные дубли

### Отображение (FR-104)

В списке — маленькая иконка-бейдж у строки лида, в карточке — отдельный блок «Похожие лиды» со ссылками на связанные записи и указанием типа совпадения (телефон/email).

### Создание вручную с проверкой дубля (FR-102)

```
POST /api/leads (создание вручную)
  → СНАЧАЛА: проверка дублей (та же функция, что в leads-intake, синхронно)
  → если найдено совпадение → 409 с телом { possibleDuplicates: [...] }, лид НЕ создаётся пока
  → UI показывает превью совпадений: «Похоже, такой лид уже есть» + карточки совпадений
  → пользователь выбирает: «Всё равно создать» (повторный запрос с флагом confirmDuplicate: true →
    лид создаётся + DuplicateFlag) ИЛИ «Открыть существующий» (переход на найденный лид, создание отменяется)
```

Это единственное место, где проверка дублей синхронна и может временно «придержать» создание лида — потому что здесь есть живой человек, который примет решение за секунды. На вебхуках (нет человека) — лид создаётся всегда, проверка идёт после (см. `leads-intake.md`).

## Комментарии, История изменений, Редактирование лида, Удаление лида — без изменений

Механика (лента, события `COMMENTED`/`LEAD_UPDATED`/`LEAD_DELETED`, права ADMIN на удаление) не менялась. В таймлайн истории добавляются строки для новых типов событий:

| `type` | Строка в таймлайне |
| --- | --- |
| `LEAD_TAKEN_IN_WORK` | «{Менеджер} взял лид в работу» |
| `LEAD_WON` | «Лид закрыт сделкой» |
| `LEAD_LOST` | «Лид закрыт отказом: {причина}» |
| `DUPLICATE_FLAGGED` | «Найден похожий лид» (со ссылкой) |

## Страница контроля (Руководитель и Администратор)

`/control` (зона `(management)`, доступна с уровня `HEAD` — не только `ADMIN`, см. `CLAUDE.md`) — без изменений в назначении самого счётчика, обновлены формулы (источник данных — те же `events`, без отдельной таблицы метрик):

```
Получено лидов      = count(Lead) где assignedToId = менеджер
Обработано в срок    = count(Lead), где есть LEAD_TAKEN_IN_WORK   ← было: LEAD_OPENED ИЛИ STAGE_CHANGED
                        в пределах применимого норматива от LEAD_CREATED
Зависло              = count(Lead) с событием LEAD_STAGE_STUCK, ещё открытых
Закрыто сделкой       = count(Lead) где closeType = WON            ← было: stage.order = последний
Закрыто отказом       = count(Lead) где closeType = LOST
Закрыто без причины   = count(Lead) где closeType = LOST И lossReasonId = null  ← структурно не должно случаться
                        (lossReasonId обязателен на уровне API), метрика — для аудита аномалий
```

**Важное разделение по уровню доступа:** `/control` — только дашборд (счётчик активности, доступен Руководителю и Администратору). Настройки контроля (тумблер, нормативы, проценты эскалации, рабочее время) — на отдельной странице `/admin/settings`, доступной только Администратору (см. `.docs/modules/app-settings.md`). Это не одна и та же страница: «смотреть, как работает команда» и «настраивать, по каким правилам это измеряется» — разные уровни ответственности.

## Серверные правила безопасности

1–8 без изменений относительно предыдущей версии (`visibilityWhere` везде, `companyId` из сессии, 404 не 403 для чужого лида в режиме `OWN`, экранирование `customFields`, удаление только ADMIN, `PATCH` не трогает данные приёма, события пишет сервер, пагинация обязательна).

9. **Повторное «Взял в работу» не создаёт второе событие.** Идемпотентная проверка перед записью `LEAD_TAKEN_IN_WORK`.

10. **Закрытие отказом без `lossReasonId` — отклоняется на сервере**, не только на клиенте.

11. **Мутации (`take`, `close`, `PATCH`, `POST`, `DELETE`) — обычные проверки видимости и роли**, без отдельного guard'а подписки — биллинга в продукте не существует. Единственная проверка доступа на входе — `Company.isBlocked` (см. `.docs/modules/auth.md`), не на уровне отдельных эндпоинтов.

12. **Проверка дублей при создании вручную — до записи в БД**, не после. На вебхуках — наоборот (после, см. `leads-intake.md`). Это единственная сознательная асимметрия в проекте между «есть человек» и «нет человека» в моменте создания.
