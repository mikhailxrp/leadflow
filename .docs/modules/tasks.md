# Модуль: Задачи (tasks)

> Спецификация создания, управления и выполнения задач по лидам. Задача привязана к лиду — при клике на задачу из любого места системы открывается карточка соответствующего лида. Открытая задача с ближайшим сроком одновременно является «следующим действием» лида (FR-170…172) — отдельной сущности под это не заводится.
> Связанные файлы: `.docs/database.md` (модели `Task`, `TaskStatus`, `EventType` `TASK_*`), `.docs/modules/leads.md` (карточка лида, «следующее действие», колонка и быстрые действия в списке), `.docs/modules/pipeline.md` (промпт при смене этапа), `.docs/modules/risk.md` (риск при отсутствии открытой задачи), `.docs/modules/admin-users.md` (управление менеджерами — исполнители задач), `.docs/modules/platform-marketer.md` (маркетолог задач не касается).
>
> **Самодостаточный источник правды.** Модуль восстановлен из полной версии (`git 104c789`) и актуализирован под текущее состояние проекта: роль «Маркетолог» (deny-by-default), корректные имена событий из enum, правило сортировки для задач без срока, отдельный `assignable`-эндпоинт. Реализуется в Phase 15 (`.docs/phases/phase-15.md`).

## Цели модуля

После завершения этого модуля:

- Администратор и менеджер создают задачи по лидам с исполнителем и сроком
- Задачи видны в карточке лида на реальном API
- У активного лида видно «следующее действие» (самая ранняя открытая задача) — в карточке и колонкой в списке лидов
- Отсутствие открытой задачи — сигнал риска (`.docs/modules/risk.md`), не ошибка
- Клик на задачу из любого места открывает карточку привязанного лида
- Менеджер меняет статус задачи (`TODO → IN_PROGRESS → DONE/CANCELLED`)
- Администратор видит и удаляет все задачи по всем лидам
- При смене этапа лида без открытой задачи — необязательный промпт «что дальше и когда»
- История изменений задачи пишется в `events`

**Не входит в модуль:**

- Глобальный список задач без привязки к лиду — out of scope MVP
- Приоритеты задач — поля нет в модели `Task`
- Уведомления о приближении дедлайна задачи — задел на будущее / Phase 17
- Шаблоны задач — out of scope MVP (типовые кнопки предзаполнения `title` — только UI-нюанс, см. ниже)

## Чем задача отличается от напоминания

|                     | Напоминание           | Задача                                    |
| ------------------- | --------------------- | ----------------------------------------- |
| Кто создаёт         | Сам себе              | HEAD+ — кому угодно, MANAGER — только себе |
| Цель                | Уведомить в срок      | Зафиксировать что нужно сделать            |
| Исполнитель         | Тот кто создал        | Назначается явно (`assignedToId`)          |
| Статус              | `PENDING` / `FIRED`   | `TODO` / `IN_PROGRESS` / `DONE` / `CANCELLED` |
| Контроль            | Нет                   | Руководитель видит все задачи              |
| Действие после срока | Автодоставка         | Ничего — исполнитель сам меняет статус     |
| Роль в UI           | Блок «Напоминания»    | Блок «Задачи» + «следующее действие»       |

## Задача как «следующее действие»

Техничка вводит требование «у каждого активного лида должен быть следующий шаг» (FR-170…172) как отдельное понятие. Решение — **не заводить новую сущность**, а считать «следующим действием» обычную открытую задачу (`status IN (TODO, IN_PROGRESS)`) лида с ближайшим сроком.

```
У лида есть открытая задача с ближайшим dueDate → она показывается как «следующее действие»
                                                  (карточка лида, колонка в списке)
У лида нет ни одной открытой задачи             → «Нет следующего действия» → сигнал риска (risk.md)
Лид закрыт (closeType != null)                  → «—», следующего действия нет по определению
```

Требование «должен быть следующий шаг» относится только к **активным** лидам. Поэтому `getNextActions` фильтрует по `lead: { closeType: null }`, а `closeLead` отменяет открытые задачи закрываемого лида (`.docs/modules/leads.md` → «Закрытие снимает открытые задачи и напоминания»). Задачи закрытых лидов не попадают и в блоки «Сегодня» — ни в «на сегодня», ни в «просроченные».

Это не меняет ничего в самой модели `Task` — меняется только то, как её читают другие модули. Отсутствие открытой задачи уже учитывается фактором риска в `lib/risk/computeRiskBatch.ts`.

### Правило «ближайшей» задачи (важно — `dueDate` nullable)

`Task.dueDate` необязательно. Порядок выбора «следующего действия» среди открытых задач лида:

```
1. Задачи с заданным dueDate — по dueDate ASC (раньше срок → выше)
2. Задачи без dueDate         — после них, по createdAt ASC
→ первая в этом порядке = «следующее действие»
```

Совпадает с `orderBy: { dueDate: 'asc' }` (в PostgreSQL `NULL` идёт последним), уже применяемым в `computeRiskBatch.ts`.

## Архитектурные решения

### 1. Задача всегда привязана к лиду

`Task.leadId` — обязательное поле, не nullable. Задачи вне контекста лида не существуют. Это намеренное ограничение MVP: задача = действие по конкретному клиенту.

### 2. Два пользователя: создатель и исполнитель

```prisma
createdById  String   // кто поставил задачу
assignedToId String   // кто должен выполнить
```

Разные поля и разные роли. Руководитель ставит задачу менеджеру. Оба видят задачу в карточке лида.

### 3. `completedAt` фиксируется автоматически

При переводе статуса в `DONE` сервер сам выставляет `completedAt = now()`. Клиент эту дату не передаёт. При переводе в `CANCELLED` — `completedAt = null`.

### 4. История через `events`

Каждое изменение задачи → событие в `events` через `lib/events.ts`. Имена — строго из enum `EventType` (`.docs/database.md`):

```
Создание                         → TASK_CREATED
Переход в DONE                   → TASK_DONE
Переход в CANCELLED              → TASK_CANCELLED
Редактирование полей / переход в TODO|IN_PROGRESS → TASK_UPDATED
Удаление                         → события нет (в EventType нет TASK_DELETED)
```

В enum нет `TASK_COMPLETED` — используется `TASK_DONE`. История задачи читается из тех же событий, что и история лида; company-side подписи — в `constants/eventLabels.ts` (`getEventLabel`).

### 5. Каскад с лидом

`onDelete: Cascade` — при удалении лида удаляются все его задачи. Задача без лида не имеет смысла.

### 6. Типовые кнопки (UI-нюанс, не схема)

Типовые задачи (позвонить, написать, отправить документ, подготовить расчёт, назначить встречу, уточнить решение) оформляются готовыми кнопками при создании — это **предзаполнение поля `title`** в форме, не новое поле и не enum `TaskType`. `title` остаётся свободным текстом.

### 7. Мутации — обычные проверки, без guard'а подписки

Создание/изменение/удаление задачи — обычные мутации с проверкой видимости и роли. Отдельного guard'а подписки не существует — биллинга в продукте нет. Единственная точка проверки доступа компании — `Company.isBlocked` при входе (`.docs/modules/auth.md`).

## Создание задачи

### Кто может создавать

- **ADMIN** — любому лиду, любому пользователю
- **HEAD** — любому лиду компании (видит все), любому пользователю
- **MANAGER** — только лидам в пределах своей видимости (`visibilityWhere`) и **только себе как исполнителю** (`assignedToId = свой userId`)

Проверка — `requireCompanyUser({ minRole: 'MANAGER' })` (маркетолог отсекается, см. «Маркетолог и задачи»).

**Назначение исполнителя — привилегия HEAD+.** Назначить задачу другому сотруднику может только руководитель и администратор. Менеджер ставит задачи исключительно себе: `GET /api/users/assignable` возвращает ему только его самого, а `POST`/`PATCH` отклоняют `assignedToId ≠ свой userId` (`403 FORBIDDEN`). Проверка — через `hasMinRole(role, 'HEAD')`. При редактировании менеджером ранее созданной задачи сохранение прежнего исполнителя не блокируется — запрет срабатывает только на *смену* исполнителя на чужого.

### Поля при создании

| Поле           | Обязательное | Описание                        |
| -------------- | ------------ | ------------------------------- |
| `title`        | Да           | Что нужно сделать (1–200)       |
| `assignedToId` | Да           | Кому назначена                  |
| `dueDate`      | Нет          | Срок выполнения (ISO datetime)  |
| `description`  | Нет          | Подробное описание (≤ 2000)     |

### Флоу

```
Кнопка «+ Создать» в блоке «Задачи» карточки лида (или «Поставить следующее действие» из строки списка)
  → модалка: title (обязательно; типовые кнопки предзаполняют title),
             assignedTo (селект из GET /api/users/assignable), dueDate (датапикер), description
  → POST /api/leads/:id/tasks
  → сервер: Task.create + writeEvent(TASK_CREATED)
  → задача появляется в карточке лида / колонке «Следующее действие» списка
```

### Промпт при смене этапа (FR-171)

```
Менеджер меняет этап лида (drag-and-drop на доске)
  → если у лида нет открытой задачи → модалка «Что делаем дальше и когда?»
  → создаёт обычную задачу через тот же POST /api/leads/:id/tasks
  → переход этапа выполняется в любом случае — это НЕ блокирующая проверка
```

## Управление задачами

### Смена статуса

Менеджер меняет статус своей задачи. Админ — любой.

```
TODO → IN_PROGRESS → DONE
                  ↘ CANCELLED (отменить)

PATCH /api/leads/:id/tasks/:taskId { status }
  → DONE       → completedAt = now()  + writeEvent(TASK_DONE)
  → CANCELLED  → completedAt = null   + writeEvent(TASK_CANCELLED)
  → TODO|IN_PROGRESS → writeEvent(TASK_UPDATED)
```

### Редактирование

`PATCH /api/leads/:id/tasks/:taskId` — изменить `title`, `description`, `dueDate`, `assignedToId` → `writeEvent(TASK_UPDATED)`.

```
Если → status === DONE или CANCELLED
То   → редактирование полей запрещено → 400 TASK_NOT_EDITABLE
```

Право на редактирование/смену статуса — **автор (`createdById`) или ADMIN**.

### Удаление

`DELETE /api/leads/:id/tasks/:taskId` — физическое удаление. Только **ADMIN**. События не пишет.

## Связь задача → лид

**Это ключевая навигационная связь модуля.** Задача всегда знает свой `leadId`. Клик на задачу из любого места → переход на карточку лида.

```typescript
// компонент TaskItem.tsx
// Если уже на карточке лида — скролл/подсветка блока задач (?taskId= в URL)
// Если в другом месте — router.push(`/leads/${task.leadId}`)
<div onClick={() => router.push(`/leads/${task.leadId}`)}>{task.title}</div>
```

При переходе с `?taskId=` карточка подсвечивает конкретную задачу (уже поддержано `LeadDetailPage`).

## Отображение задач

### Блок «Задачи» в карточке лида

Правая колонка карточки, рядом с блоком «Напоминания». Данные — `GET /api/leads/:id/tasks`.

```
┌─────────────────────────────────────────────┐
│ Задачи                           [+ Создать]  │
│                                              │
│ ○ Отправить КП                              │
│   Алексей · до 15 июня            [TODO]    │
│ ◉ Провести демо                             │
│   Мария · до 20 июня         [IN_PROGRESS]  │
│ ─ Выполненные и отменённые (свёрнуто) ─     │
│ ✓ Согласовать договор  · выполнено 10 июня  │
└─────────────────────────────────────────────┘
```

**Сортировка:** активные (`TODO`/`IN_PROGRESS`) сверху по правилу «ближайшей задачи» (см. выше); просроченные (`dueDate < now`, статус активный) выделены; выполненные и отменённые — внизу, свёрнуто.

### Колонка «Следующее действие» в списке лидов

В таблице `/leads` — read-only колонка: название + срок ближайшей открытой задачи, либо «Нет следующего действия» (визуально как предупреждение), либо «—» у закрытого лида. Данные — батч-резолвер `lib/tasks/getNextActions.ts` (`leadIds → { taskId, title, dueDate }`), один запрос на страницу.

### Быстрые действия из строки списка

Из строки, без открытия карточки (только `actor.actor === 'user'`):

- **«Поставить следующее действие»** — компактная модалка создания задачи (переиспользует `AddTaskModal`)
- **«Изменить срок»** — если есть открытая задача → правит её `dueDate`; если нет → предлагает сначала поставить следующее действие

### Бейджи статусов

| Статус        | Цвет                    |
| ------------- | ----------------------- |
| `TODO`        | Серый                   |
| `IN_PROGRESS` | Синий `#3B82F6`         |
| `DONE`        | Зелёный `#10B981`       |
| `CANCELLED`   | Серый, зачёркнут        |

## Маркетолог и задачи

Маркетолог (виртуальный actor `marketer` в company-сессии, `.docs/modules/platform-marketer.md`) **задач не касается**: просмотр лидов и воронки у него read-only + квалификация, задач в его allow-list (`constants/marketerAccess.ts`) нет.

- Все эндпоинты задач и `GET /api/users/assignable` — `requireCompanyUser({ minRole: 'MANAGER' })` → маркетологу `403` (deny-by-default).
- Блок «Задачи» в карточке и быстрые действия в списке рендерятся только при `actor.actor === 'user'` — маркетолог их не видит (как `ReminderBlock`/`LeadComments`).

## API-эндпоинты

| Метод  | Путь                             | Назначение                       | Auth    | Право                    |
| ------ | -------------------------------- | -------------------------------- | ------- | ------------------------ |
| GET    | `/api/leads/:id/tasks`           | Список задач по лиду             | Session | Видимость по роли (не marketer) |
| POST   | `/api/leads/:id/tasks`           | Создать задачу                   | Session | MANAGER+ (не marketer)   |
| PATCH  | `/api/leads/:id/tasks/:taskId`   | Изменить / сменить статус        | Session | Автор или ADMIN          |
| DELETE | `/api/leads/:id/tasks/:taskId`   | Удалить задачу                   | Session | ADMIN only               |
| GET    | `/api/users/assignable`          | Активные пользователи `{id,name}` для селекта исполнителя | Session | MANAGER+ (не marketer) |

### `POST /api/leads/:id/tasks`

**Request:**
```json
{
  "title": "Отправить коммерческое предложение",
  "assignedToId": "clxxx",
  "dueDate": "2026-06-20T10:00:00Z",
  "description": "Включить тариф Про и примеры кейсов"
}
```

**Response 200:**
```json
{
  "id": "clyyy",
  "title": "Отправить коммерческое предложение",
  "status": "TODO",
  "assignedTo": { "id": "clxxx", "name": "Алексей" },
  "dueDate": "2026-06-20T10:00:00Z",
  "leadId": "clzzz",
  "createdAt": "2026-06-11T08:00:00Z"
}
```

**Response 400 (невалидный исполнитель):** `{ "success": false, "error": "ASSIGNEE_INVALID" }`

### `PATCH /api/leads/:id/tasks/:taskId`

**Request (смена статуса):** `{ "status": "DONE" }`
**Request (редактирование):** `{ "title": "...", "dueDate": "...", "assignedToId": "..." }`

**Response 400 (задача в финальном статусе):** `{ "success": false, "error": "TASK_NOT_EDITABLE" }`

## Серверные правила безопасности

1. **`companyId` из сессии.** Лид проверяется по `companyId` пользователя — нельзя создать задачу на лид чужой компании.

2. **Видимость задач через видимость лида.** `visibilityWhere` из `.docs/modules/leads.md` применяется к лиду прежде чем отдать/менять его задачи — менеджер в режиме `OWN` не видит задачи чужого лида (`404`, не `403`).

3. **Создание — MANAGER+, редактирование/статус — автор или ADMIN, удаление — ADMIN.** Проверка роли — через `hasMinRole()`, не перечислением значений enum.

4. **Маркетолог — deny-by-default.** Все эндпоинты задач и `assignable` используют `requireCompanyUser({ minRole: 'MANAGER' })` → маркетологу `403`; блок и быстрые действия скрыты при `actor.actor !== 'user'`.

5. **`completedAt` — только сервер.** Клиент дату выполнения не передаёт — сервер ставит сам при `status = DONE`.

6. **`assignedToId` — активный пользователь той же компании.** Нельзя назначить задачу заблокированному пользователю или пользователю чужой компании (`400 ASSIGNEE_INVALID`). Список для селекта — `GET /api/users/assignable` (MANAGER+), т.к. `GET /api/users` доступен только HEAD+.

7. **Назначать других — только HEAD+.** Менеджер ставит задачи только себе (`assignedToId = свой userId`), иначе `403 FORBIDDEN`. `GET /api/users/assignable` отдаёт менеджеру только его самого; `POST`/`PATCH` сверяют `hasMinRole(role, 'HEAD')`. `PATCH` блокирует лишь *смену* исполнителя на чужого — сохранение прежнего (в т.ч. из задач до этого правила) проходит.

8. **Редактирование `DONE`/`CANCELLED` запрещено** (`400 TASK_NOT_EDITABLE`) — завершённая задача финальна.

9. **События пишет только сервер** через `lib/events.ts`, корректными типами из enum (`TASK_CREATED`/`TASK_DONE`/`TASK_UPDATED`/`TASK_CANCELLED`); удаление события не пишет.

10. **Промпт при смене этапа не блокирует переход.** Смена `stageId` выполняется независимо от того, создал ли пользователь задачу в промпте.
